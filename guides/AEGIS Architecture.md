PART 1 — System Foundation and Security Architecture

Section 1 — System Identity, Design Philosophy, and Architecture Overview
What AEGIS Is
AEGIS is a production-grade enterprise AI platform built specifically for Sona Comstar's internal SAP operations. Its purpose is to transform how Sona Comstar employees interact with SAP knowledge. When a warehouse employee encounters an error while creating a delivery, when a finance executive cannot determine why a posting is blocked, or when a procurement officer needs to understand the correct procedure for creating a scheduling agreement — instead of calling IT, raising a ticket, or searching through disorganised documentation folders, the employee opens AEGIS in their browser and receives a specific, accurate, step-by-step answer grounded in Sona Comstar's own verified SAP documentation.
AEGIS does not generate answers from a language model's general training memory. Every claim in every answer it produces is verified against retrieved documentation before delivery. Every answer carries a confidence score derived from three independent quality signals. Every source is cited by name and verification date. If the system cannot find sufficient documentation to answer reliably, it says so explicitly and creates a support ticket rather than delivering a confident-sounding but ungrounded answer.
The system serves three user groups. Employees use the web-based chat interface to ask questions and receive help. The IT admin uses the admin portal to manage the knowledge base, review system quality, and handle escalations. The engineering team monitors system health through observability dashboards.
The Name
AEGIS — Adaptive Enterprise Grade Intelligence System. In classical mythology the aegis is an impenetrable divine shield, carried into battle to protect those who bear it. The name reflects the two defining properties of this architecture: enterprise-grade security at every layer protecting both the system and its data, and intelligent adaptive behaviour that matches the right processing depth to every query. An aegis does not just defend — it also projects strength. AEGIS does not just protect Sona Comstar's SAP knowledge — it makes that knowledge accessible and actionable.
Six Core Design Principles
Every architectural decision in AEGIS traces back to one of six design principles. These are not aspirational statements — they are engineering constraints that determined component selection, component interaction patterns, and tradeoff resolutions throughout the design process. Understanding them explains why the architecture looks the way it does.
Principle 1 — Quality Over Speed, Balanced Not Traded
Answer accuracy and completeness take priority over response latency. A wrong answer delivered in 5 seconds is worse than a correct answer delivered in 35 seconds, particularly in an SAP context where acting on incorrect steps can corrupt transactions, block business processes, or produce compliance violations. However, this principle is not a licence to ignore latency entirely. The architecture uses streaming, caching, adaptive pipeline depth, and concurrent validation to minimise latency at every point. When a genuine conflict exists between quality and speed, quality wins. When both can be served without compromise, both are served.
Principle 2 — Zero Trust Security Without Exception
No service in AEGIS trusts any other service simply because they share the same server, the same Docker network, or the same deployment configuration. Every service-to-service call is mutually authenticated using short-lived certificates. Every secret is dynamic, generated on demand, and expires automatically. Every action by every component is logged in an append-only audit trail. This principle holds for an internal enterprise system used only by Sona Comstar employees because internal systems are targeted by compromised credentials, insider threats, and lateral movement attacks at significantly higher rates than external-facing systems. Building zero trust from the start costs a modest amount of engineering effort. Retrofitting it onto an insecure foundation costs enormously more and often never achieves the same standard.
Principle 3 — Grounded Generation, Never Open Memory Recall
The language model generating answers is never permitted to answer from its own training memory about Sona Comstar's SAP configuration, error patterns, procedures, or policies. Every factual claim in every answer must be traceable to a specific chunk of specific documentation in the AEGIS knowledge base. The three-tier validation system enforces this mechanically — ungrounded claims are detected, flagged, and either corrected through targeted regeneration or cause the answer to be suppressed before delivery. This principle is the reason AEGIS is trustworthy in an enterprise context where answers have real operational consequences.
Principle 4 — Adaptive Intelligence, Not Fixed Pipeline Depth
Different queries deserve different amounts of retrieval effort, model reasoning depth, and validation scrutiny. A question about the exact T-code for tax code creation can be answered with a single registry lookup and a 7B model. A question about why an MM purchase order is causing an FI posting failure across two modules requires multi-query deep retrieval, a 32B model with chain-of-thought reasoning, and full three-tier validation. AEGIS routes each query to the appropriate processing depth automatically through its retrieval mode system and tier selection logic. This means simple queries are fast and complex queries are thorough. The opposite — all queries at maximum depth — would produce uniform 90-second response times for a question whose answer is a single T-code.
Principle 5 — Operational Simplicity for the Team That Runs It
AEGIS is operated by a small intern team with limited production operations experience. Every component is chosen with this reality in mind. Where a simpler tool exists that meets requirements, it is chosen over a more powerful but operationally complex alternative. ARQ over Celery. Docker Compose over Kubernetes. Two focused Redis instances over Redis Cluster. Prometheus and Grafana over a full ELK stack. Where operational complexity is unavoidable because it is genuinely necessary — Vault for dynamic secrets, mTLS for service authentication — it is isolated, clearly documented, and given explicit operational runbooks.
Principle 6 — Honest Uncertainty Communication
AEGIS never presents an uncertain answer as if it were certain. Every answer carries a confidence indicator derived from three independent quality signals. When confidence is below a medium threshold, the answer is qualified with a disclaimer. When confidence is below a minimum threshold, the system declines to answer and escalates. This honesty is a feature that builds employee trust over time — employees who consistently see that AEGIS's green-confidence answers are accurate and its amber-confidence answers need verification will rely on the system appropriately and will not be misled by an overconfident wrong answer.
The Six-Zone Architecture
AEGIS organises all components into six functional zones. Each zone has a clearly defined and bounded responsibility. Communication between zones follows strict rules — certain zones cannot communicate directly and must go through an intermediary. This organisation is not cosmetic — it reflects genuine security boundaries, dependency hierarchies, and operational separation.
Zone A — Edge, Security and Identity
The perimeter. All employee and admin requests enter the system through Zone A. All responses exit through it. Zone A handles TLS termination, authentication token validation, token revocation checking, rate limiting, and the Enterprise Content Governance Layer. It also runs the secret management infrastructure (Vault) and identity provider (Keycloak) that every other zone depends on. A request that does not pass Zone A's checks never reaches any other zone.
Zone B — Orchestration, Pipeline and State Management
The coordinator. Zone B receives validated requests from Zone A, loads the employee's conversation session state, coordinates the sequence of operations across Zone C and Zone D, manages background task dispatch, implements circuit breaking so failures degrade gracefully, and handles the streaming response delivery mechanism. Zone B knows what order things happen in and what to do when any step fails.
Zone C — AI Services
The intelligence. Zone C contains five service clusters: the Query Intelligence Layer (understanding the query), the Vision Service (reading screenshots), the Adaptive Retrieval Engine (finding relevant documentation), the Reasoning Service (generating the answer), and the Validation and Quality Assurance Engine (verifying the answer). Zone C is entirely stateless — every service receives input, produces output, and has no memory beyond what Zone B passes explicitly.
Zone D — Data Architecture
The memory. Zone D contains all persistent storage: four Qdrant collections for vector search, OpenSearch for keyword search, PostgreSQL for structured data, and two Redis instances for session management and background task queuing. Zone D is not directly accessible from Zone A or from employee browsers — all data access goes through Zone B or Zone C services.
Zone E — Knowledge Management and Administration
The operational surface. Zone E contains the eleven-stage Document Ingestion Pipeline and the Admin Portal. The IT admin interacts with AEGIS through Zone E — uploading documents, managing the Known Patterns Registry, updating the Config Snapshot, viewing quality metrics, reviewing escalations, and examining the employee audit trail.
Zone F — Observability and LLMOps
The awareness layer. Zone F collects metrics from every other zone, provides the dashboards the team uses to monitor system health and AI answer quality, and surfaces the intelligence that guides ongoing improvement.

Demo vs Production Configuration
The demo and production deployments share the same codebase, the same architectural design, and the same security model. They differ in four areas only — inference hardware and model scale, authentication flow security level, high availability configuration, and ticket integration.
In the demo configuration, all inference runs on the existing Sona Comstar server (Intel Xeon E-2278G, 16 threads, 64GB RAM) using three Ollama instances. The main generation model is Qwen2.5-32B-Instruct at Q4_K_M quantization. Authentication uses ROPC flow (acceptable for isolated internal demo). No Qdrant replica. Mock ticket pipeline. Total memory usage approximately 52GB.
In the production configuration, inference moves to a private cloud GPU server accessed through Azure Private Endpoints or AWS PrivateLink — no public internet exposure of SAP query data. vLLM replaces Ollama. Main generation upgrades to Qwen2.5-72B-Instruct at AWQ quantization. Authentication uses Authorization Code Flow with PKCE. Qdrant replica returns. BGE-M3 sparse retrieval returns. Vault uses cloud KMS auto-unseal. Real IT ticketing system integration. Everything else — retrieval pipeline, validation framework, session management, data schemas, security model, observability — is identical.

Section 2 — Zone A: Edge, Security and Identity
Overview and Security Philosophy
Zone A is the outermost boundary of AEGIS. It is the only zone that accepts connections from the corporate intranet, and the only zone through which responses exit. Every component in Zones B through F is completely hidden behind Zone A and unreachable without passing through it first.
The security model in Zone A is not perimeter defence — it is zero trust at the entry point. Even after an employee passes authentication at the Nginx boundary, their request is re-verified inside FastAPI. Even after a service receives a valid request from another service, the receiving service verifies the mTLS certificate of the calling service before processing. Defence is layered, redundant, and assumes that any single layer may be bypassed.
Zone A has five distinct security responsibilities that operate in sequence on every inbound request: TLS encryption enforcement (Nginx), identity authentication (Keycloak token verification), behavioural rate limiting (Redis-backed counters), content governance (rule-based input analysis), and secret provision (Vault dynamic credential management). These responsibilities are handled by different components, but they form a single coherent security pipeline.
Nginx — The Sole Entry Point
Nginx is the first and only process that accepts connections from the corporate intranet. Nothing in AEGIS has a network route accessible from outside Zone A except through Nginx. This is enforced by Docker network topology described in the deployment section.
TLS Configuration
Nginx listens on two ports. Port 80 accepts HTTP connections and immediately issues a 301 permanent redirect to HTTPS — no HTTP request ever reaches the application layer. Port 443 accepts HTTPS connections. TLS is configured to accept only TLS 1.3 — TLS 1.2 and all earlier versions are disabled in the Nginx SSL configuration. The cipher suite list is restricted to AEAD ciphers (AES-256-GCM and ChaCha20-Poly1305) that provide both encryption and message authentication. The TLS certificate for the AEGIS domain is stored on the Nginx container and renewed through a standard certificate renewal procedure.
Nginx terminates TLS at the boundary — it decrypts the incoming HTTPS request and forwards plain HTTP to FastAPI over the internal Docker application network. The internal forwarding is not TLS-encrypted because it travels entirely within Docker's internal network on the same host. The security boundary at this layer is Docker network isolation, not transport encryption. The mTLS layer described later handles service-to-service authentication for all cross-service calls.
Request Validation Before Routing
Before forwarding any request to FastAPI, Nginx performs four checks that reject invalid requests immediately without ever reaching application code.
Request size enforcement rejects any request whose body exceeds 50 megabytes. This applies to all endpoints. For the file upload endpoint, an additional per-field limit of 30 megabytes is enforced on the file attachment field. This prevents employees from accidentally or deliberately uploading enormous files that would exhaust server memory during ingestion.
Content-Type validation checks the Content-Type header against the permitted values for each endpoint. The chat endpoint accepts only application/json. The document upload endpoint accepts only multipart/form-data. The admin API endpoints accept application/json. Any mismatch between the Content-Type header and the endpoint's expected type returns a 400 Bad Request before FastAPI processes the request.
HTTP method enforcement rejects requests using disallowed HTTP methods on each endpoint. The chat endpoint accepts only POST. The document upload endpoint accepts only POST. Admin GET endpoints accept only GET. Any method mismatch returns a 405 Method Not Allowed.
Defined path routing means that any request to a URL path that does not match a defined API route receives a 404 immediately from Nginx without being forwarded. This prevents path traversal probing and ensures the application code never processes requests to undefined endpoints.
Rate Limiting
Nginx maintains a rate limiting state using a shared memory zone. Each employee is identified by the sub claim extracted from their JWT token in the Authorization cookie. The rate limit is 60 requests per minute per employee, with a burst capacity of 10 additional requests. The burst capacity handles legitimate patterns where an employee submits a follow-up message at the same moment they receive a response.
When a rate limit is exceeded, Nginx returns a 429 Too Many Requests response with a Retry-After header specifying when the employee can retry. The FastAPI application never receives the request — rate-limited responses are entirely handled at the Nginx layer with zero application processing overhead.
The rate limit state is maintained in an Nginx shared memory zone rather than Redis because this check must occur before any application connection is opened. The shared memory zone is process-local to Nginx and does not persist across Nginx restarts, which is acceptable — rate limit counters reset on the rare occasions Nginx restarts.
Keycloak — Identity and Authentication
Keycloak is the OIDC/OAuth2 identity provider for AEGIS. It is the single source of truth for who is allowed to use the system. It manages employee credentials, issues cryptographic tokens that prove identity, and provides the mechanism for immediate credential revocation.
Database Backend
Keycloak runs with PostgreSQL as its data backend. The Keycloak container connects to the AEGIS PostgreSQL instance using a dedicated keycloak database schema. This is critical for operational stability: Keycloak's default Docker configuration uses H2, an embedded Java database stored in memory with optional file persistence. H2 is explicitly not recommended for production by Keycloak's own documentation. If the Keycloak container restarts without perfectly configured H2 file persistence, all user accounts, client registrations, and configuration are lost — the IT admin would need to manually recreate every employee account. By using PostgreSQL, Keycloak's data is stored durably in the same PostgreSQL instance that backs the rest of AEGIS, protected by the same backup strategy.
Realm Configuration
Keycloak is configured with a single isolated realm called aegis-realm. A realm in Keycloak is a complete identity namespace — it has its own user database, its own token settings, its own registered client applications, and no connection to any other Keycloak realm. The aegis-realm is entirely isolated from any other system.
Within the realm, two client applications are registered. The aegis-chat client represents the employee-facing frontend. The aegis-admin client represents the admin portal. Each client has different permission scopes — employees can access employee-level API endpoints, while IT admins can additionally access administrative endpoints. Both clients share the same Keycloak instance and realm.
Two user roles exist: employee and it-admin. When the IT admin creates an account for a new user through the Keycloak admin console, they assign one of these roles. The role is embedded as a claim in the issued JWT and checked by FastAPI's authentication middleware on every request.
The Authentication Flow — Demo Configuration (ROPC)
The demo configuration uses the OAuth2 Resource Owner Password Credentials flow. This flow is appropriate for the demo because the system is completely isolated from Sona Comstar's production network and is accessed only by the IT team during evaluation. It will be replaced with Authorization Code Flow + PKCE before production deployment.
When an employee opens AEGIS and has no valid session, the Next.js frontend displays a login form. The employee enters their username and password. The frontend sends these credentials directly to Keycloak's token endpoint. Keycloak receives the credentials, looks up the user in the aegis-realm user database, verifies the bcrypt password hash, and if authentication succeeds, issues two tokens.
The access token is a JSON Web Token signed with Keycloak's RSA-256 private key. It is a compact base64-encoded string containing three parts: a header describing the signing algorithm, a payload containing the claims, and a cryptographic signature. The payload claims include: sub (the employee's unique user ID), role (either employee or it-admin), iat (issued-at timestamp), exp (expiry timestamp, set to 15 minutes from issue), jti (a unique token identifier used for revocation), aud (the intended audience, set to the client ID), and iss (the issuer, set to the Keycloak realm URL). The 15-minute TTL means even if an access token is intercepted, it is valid for at most 15 minutes.
The refresh token is an opaque random string stored by Keycloak in its PostgreSQL database. It is not a JWT — it contains no claims that can be read by the application. Its only purpose is to obtain a new access token when the current one expires. Refresh tokens have an 8-hour TTL appropriate for a manufacturing shift cycle — an employee who starts work at the beginning of a shift can use AEGIS throughout the shift without re-authenticating.
Both tokens are stored by the frontend in an HttpOnly, Secure, SameSite=Strict cookie. HttpOnly means the cookie cannot be read by JavaScript running in the browser — only the browser itself can include it in requests. This completely eliminates token theft through XSS attacks. Secure means the cookie is only transmitted over HTTPS connections. SameSite=Strict means the cookie is never sent with cross-origin requests, eliminating CSRF attacks.
Token Refresh — Transparent Session Continuation
Every 12 minutes (3 minutes before the 15-minute access token expires), the Next.js frontend automatically sends a silent background request to Keycloak's token endpoint using the refresh token. Keycloak validates the refresh token, checks that it has not been revoked or expired, and issues a new access token. The new access token replaces the previous one in the cookie. The employee experiences no interruption — there is no re-authentication prompt, no visible delay, and no awareness that token renewal occurred.
If the refresh token itself expires (8 hours of inactivity), the silent refresh fails. The frontend detects the 401 response and displays the login form. The employee authenticates fresh. This is correct behavior — an 8-hour session TTL is appropriate for Sona Comstar's shift structure.
Token Revocation — Immediate Effect on Security Events
When the IT admin needs to revoke an employee's access — due to termination, role change, or suspected account compromise — the following sequence occurs:
The IT admin clicks the revocation action in the Keycloak admin console or in the AEGIS admin portal's employee management interface. Keycloak immediately invalidates the refresh token in its PostgreSQL database. Keycloak simultaneously fires a webhook to a registered endpoint at the AEGIS FastAPI application, passing the jti value of the employee's current access token.
FastAPI receives the webhook, extracts the jti, and adds it to the revoked_tokens set in Redis Instance 1. The set entry has a TTL equal to the access token's remaining valid lifetime — after the token would have naturally expired anyway, the revocation entry is no longer needed and is automatically removed by Redis.
On every subsequent request from any employee, FastAPI's authentication middleware performs a single Redis SISMEMBER command — checking whether the request's JWT jti value is present in the revocation set. This operation is O(1) and takes under 0.1 milliseconds. If the jti is found, the request is rejected with a 401 Unauthorized immediately, regardless of whether the token has technically not expired yet. Revocation takes effect on the very next request after the IT admin acts.
Token Verification in FastAPI
When a request arrives at FastAPI from Nginx, the authentication middleware runs before any business logic. It extracts the JWT from the HttpOnly cookie. It verifies the JWT signature using Keycloak's RSA public key, which was fetched from Keycloak's JWKS endpoint at FastAPI startup and cached in memory. It checks the exp claim to ensure the token has not expired. It checks the jti against the Redis revocation set. It validates the aud claim matches the expected client ID. If all checks pass, the middleware extracts the sub (user ID) and role claims and stores them in the request context for downstream handlers to use. If any check fails, a 401 Unauthorized is returned immediately.
Mutual TLS Service Mesh — Authenticated Service-to-Service Communication
The mTLS service mesh is the mechanism that prevents a compromised internal service from calling other services without authorisation. In standard TLS, only the server proves its identity to the client. In mutual TLS, both sides prove their identity — the client presents a certificate proving it is who it claims to be, and the server verifies that certificate before accepting the connection.
What This Achieves
If an attacker somehow gained code execution inside the DeBERTa NLI service container, they would be able to make network calls from within that container. Without mTLS, those calls could reach Qdrant, PostgreSQL through PgBouncer, or any other service on the internal data network. With mTLS, the attacker's calls would be rejected because the compromised DeBERTa container does not hold a valid certificate authorised to communicate with those services. The mTLS configuration defines exactly which services are allowed to communicate with which other services, and enforces this at the certificate level.
Vault as Internal Certificate Authority
Vault's PKI Secrets Engine acts as the Certificate Authority for AEGIS. At initial Vault setup, the PKI engine generates a root CA private key and a root CA certificate. This root CA certificate is distributed to all services as their trust anchor — when a service receives a certificate from another service, it verifies the certificate was signed by this root CA.
Vault's PKI engine can issue leaf certificates on demand, signed by the root CA, with any specified TTL. For AEGIS, leaf certificates have a 24-hour TTL — they expire one day after issuance.
How a Service Obtains Its Certificate
When any AEGIS service starts up (for example, the FastAPI application), it first authenticates to Vault using AppRole authentication (described in the Vault section below). Once authenticated, it calls Vault's PKI endpoint requesting a certificate with the service's common name (for example, aegis-fastapi.internal). Vault generates a private key and certificate for the service, signs the certificate with the root CA, and returns both to the service. The service stores the certificate and private key in memory — they are never written to disk.
The service then uses these credentials for all outbound service-to-service connections: presenting its certificate to prove its identity, and verifying incoming certificates against the root CA to confirm the identity of services calling it.
Certificate Rotation Without Downtime
Each service runs a background renewal coroutine that checks the remaining lifetime of its current certificate every hour. When fewer than 6 hours remain on the current certificate (which has a 24-hour lifetime, so renewal begins after 18 hours), the service requests a new certificate from Vault. It receives the new certificate while the old one is still valid. The service then atomically swaps to the new certificate for all new connections while allowing existing connections to complete with the old certificate. This ensures certificates are always fresh and rotation happens transparently with zero service interruption.
Which Services Communicate With Which
Nginx to FastAPI: Nginx does not use mTLS for this connection — it uses plain HTTP over the internal Docker application network. Nginx validates JWT tokens directly rather than establishing an mTLS connection to FastAPI. This is correct because Nginx is the network boundary where TLS terminates.
FastAPI to all other Zone C services (Ollama instances, DeBERTa, BGE Embedding): mTLS required.
FastAPI to all Zone D services (Qdrant, OpenSearch, PgBouncer/PostgreSQL, Redis Instance 1, Redis Instance 2): mTLS required.
FastAPI to Vault: mTLS required (after initial AppRole authentication).
ARQ worker to Redis Instance 2: mTLS required.
ARQ worker to Ollama Instance 3 (vision): mTLS required.
ARQ worker to Qdrant (for vision DiagnosticObject cache writes): mTLS required.
All Zone F services (Prometheus scraping metrics endpoints): mTLS required — Prometheus presents a certificate when scraping, services verify it.
HashiCorp Vault — Dynamic Secret Management
Vault is the most architecturally foundational security component in AEGIS. Its purpose is to ensure that no static secret — no hardcoded password, no permanent API key, no long-lived database credential — exists anywhere in the system. Every secret any service needs is generated on demand, has a short lifespan, and expires automatically.
Deployment Mode for Demo — Development Mode
Vault runs in development mode for the AEGIS demo. Development mode is started with the command vault server -dev and is configured with a fixed root token specified in the Docker Compose environment. In development mode, Vault automatically unseals itself on every start — there are no Shamir key shares, no manual unseal ceremony, and no possibility of the system staying down after a restart because Vault is sealed.
Development mode stores all data in memory. When Vault restarts, its internal state is lost — all issued dynamic credentials expire immediately, all PKI certificates are no longer valid, and all services must re-authenticate and obtain new credentials. This sounds like a problem but is not, because all AEGIS services are designed to authenticate to Vault at startup and obtain fresh credentials — this is exactly what they do on every start regardless of whether Vault restarted. The only operational impact of a Vault restart is that all other services must restart as well, since their existing credentials become invalid. In practice, the entire Docker Compose stack restarts together, so this is not an issue.
Development mode also exposes Vault on its API port without TLS. For the isolated internal demo environment, this is acceptable. For production, Vault uses TLS and cloud KMS auto-unseal.
AppRole Authentication — How Services Prove Their Identity to Vault
AppRole is Vault's authentication method for machine-to-machine authentication. It works as follows:
Each AEGIS service has a Role ID — a static identifier set in the Docker Compose environment variables. The Role ID is not secret — it is an identifier, not a credential. It is equivalent to a username.
The Secret ID is the actual credential. For the demo with Vault in development mode, each service's Secret ID is a static value set in the Docker Compose environment. For production with Vault in full mode, Secret IDs are wrapped tokens — Vault wraps the Secret ID in a cryptographic envelope and the service must unwrap it to access the real Secret ID. The wrapping token is single-use, ensuring that if it is intercepted during deployment, it has already been consumed.
When a service starts, it presents its Role ID and Secret ID to Vault's AppRole login endpoint. Vault verifies both, checks that the Secret ID has not been used already or expired, and issues a Vault token — a short-lived credential the service uses for all subsequent Vault API calls. This Vault token has a 6-hour TTL and is renewed by the service's background renewal coroutine before expiry.
Dynamic PostgreSQL Credentials
When FastAPI needs to establish a database connection, it does not use a hardcoded username and password. Instead, it calls Vault's Database Secrets Engine requesting credentials for the aegis-operational-role lease. Vault connects to PostgreSQL using its own administrative credentials, executes a CREATE USER statement with a randomly generated username and password, grants the new user the permissions defined in the aegis-operational-role (read and write on operational tables, read-only on analytical tables, no access to Keycloak schema), and returns the username and password to FastAPI. These credentials have a 1-hour TTL.
After 1 hour, Vault automatically executes DROP USER on that credential pair in PostgreSQL. The service's renewal coroutine requests new credentials before the 1-hour expiry and PgBouncer's connection configuration is updated. If an attacker gains access to the running FastAPI process and extracts the database credentials, those credentials expire within at most 1 hour and are then permanently unusable.
Vault Transit Secrets Engine — Field-Level Encryption
Vault's Transit Secrets Engine is a cryptographic service-as-an-API. When AEGIS needs to store sensitive data — specifically, conversation summaries containing SAP configuration values, and the Config Snapshot table values themselves — it does not encrypt the data using a locally held key. Instead, it calls Vault's Transit endpoint with an encrypt request. Vault encrypts the data using AES-256-GCM and returns ciphertext. AEGIS stores the ciphertext in PostgreSQL. The actual encryption key never leaves Vault.
When AEGIS needs to read the sensitive data back, it calls Vault's Transit endpoint with a decrypt request. Vault decrypts and returns plaintext. Someone who gains direct read access to the PostgreSQL database sees only ciphertext in the sensitive fields.
PKI Secrets Engine — Certificate Issuance
The PKI Secrets Engine manages the internal Certificate Authority described in the mTLS section. When any service requests a certificate, it calls Vault's PKI endpoint with its desired common name, an optional list of Subject Alternative Names, and the requested TTL (24 hours for all AEGIS service certificates). Vault generates the private key and certificate internally, signs the certificate with the root CA, and returns both to the requesting service. All certificate generation, signing, and private key management happens within Vault — services never manage their own PKI.
Enterprise Content Governance Layer
The original MERIDIAN architecture had a 5-layer LLM-based guardrail designed for consumer AI safety — adversarial prompt injection detection, jailbreak prevention, and content policy enforcement against anonymous internet users. AEGIS replaces this with an Enterprise Content Governance Layer designed specifically for the actual threat model of an authenticated internal corporate system.
The Governance Layer has three components. The first two are synchronous — they run in the request path and add latency when they fire. The third is entirely asynchronous — it adds zero latency to any request.
Component 1 — Input Governance (synchronous, under 5ms)
Input Governance runs after JWT authentication passes and before any business logic executes. It performs three sequential checks, each taking under 2 milliseconds.
Schema validation confirms the request body matches the expected JSON or multipart structure for the endpoint being called. A chat request must have a message string field and an optional session_id field. A file upload must have a file field and a document_type field. Any structural deviation returns a 400 Bad Request immediately. This check catches malformed requests before any parsing or processing begins, preventing unexpected data from reaching downstream components.
File type validation applies only to requests with file attachments. The first 12 bytes of the uploaded file — the magic bytes — are read and compared against the signatures for accepted file types: JPEG (FF D8 FF), PNG (89 50 4E 47), DOCX (50 4B 03 04, the ZIP signature used by all modern Office documents), and PDF (25 50 44 46). A file that claims to be a PNG in its Content-Disposition header but has PDF magic bytes is rejected regardless of the claimed extension. This prevents file type confusion attacks where an attacker names a malicious file with a safe extension.
SAP-context injection pattern detection scans the text content of chat messages against a set of patterns specific to the AEGIS threat model. These are not generic consumer LLM jailbreak patterns. They are patterns that have no legitimate SAP helpdesk use case but that could be used to manipulate AEGIS: attempts to override or reveal the system prompt (phrases like "ignore your previous instructions", "repeat your system prompt word for word", "disregard your context"), attempts to extract SAP credentials or system information ("tell me the database username", "what are the credentials for"), and attempts to force the model to act outside its SAP helpdesk role ("act as an unrestricted AI", "pretend you have no restrictions"). This check is purely rule-based — a set of compiled regex patterns against lowercased message text — taking under 2 milliseconds with no model inference.
If all three checks pass, the request is marked as governance-approved and proceeds to FastAPI's business logic handlers.
Component 2 — Output Governance (synchronous, under 5ms)
Output Governance runs after the language model completes generation and before any output reaches the employee's browser. Because AEGIS uses concurrent streaming (described in Zone C-4), Output Governance runs sentence-by-sentence during generation rather than as a final blocking check. For each sentence completed by the generation model, a rapid scan runs before that sentence is forwarded through the streaming pipeline.
The scan checks the completed sentence against five pattern categories. Internal infrastructure patterns look for Docker network IP addresses in the 172.16.0.0/12 range and container hostnames (patterns like aegis-, qdrant-, postgres-). JWT token patterns look for the three-part base64-dot-base64-dot-base64 structure that identifies a JWT. Vault path patterns look for strings beginning with /v1/secret/, /v1/pki/, or /v1/database/. System prompt fragments look for the exact opening phrases of the AEGIS system role instructions — if the model echoes system prompt content, this pattern fires. SAP credential patterns look for strings resembling SAP usernames, authorisation object names, or configuration table primary keys that should never appear in employee-facing responses.
If any pattern matches, the matched content is redacted from the sentence before streaming it forward, and a flag is set in the validation context indicating that Output Governance triggered. This flag causes the Validation Engine to treat the answer as having a Tier 1 failure and initiate targeted regeneration after the current streaming sequence completes.
Component 3 — Behavioral Audit (asynchronous, zero latency impact)
Behavioral Audit runs entirely through the ARQ background task queue — it adds no latency to any request. After every request completes, regardless of whether it succeeded or failed, an ARQ audit task is queued. The task writes a complete audit record to the audit_log PostgreSQL table.
The audit record contains: timestamp, user_id_hash (a one-way SHA-256 hash of the employee's sub claim — the IT admin can see that a session occurred but cannot trivially link it to a specific employee without a separate reverse-lookup), session_id, trace_id (the unique identifier that flows through all components for this request), request_type (chat/upload/admin), governance_trigger_flags (which if any Input or Output Governance checks fired), validation_score (final AEGIS ValidationScore for this answer), model_tier (1, 2, or 3), retrieved_document_ids (list of document IDs included in the retrieval context), confidence_badge (green/amber/none), and feedback_signal (positive/negative/none, updated when the employee submits feedback within the session).
The audit_log table is append-only — the database role used by the application has INSERT permission but no UPDATE or DELETE permission on this table. This means audit records cannot be tampered with after creation, providing a tamper-evident record of all system activity.
Docker Network Topology
AEGIS uses five Docker networks, each representing a strict security boundary. A service can only communicate with another service if both are members of the same network. Docker enforces this at the network layer — packets cannot cross between networks regardless of IP routing configuration.
nexus-public network contains only Nginx. This is the network interface that accepts employee connections from the corporate intranet. No other service has a network interface on this network — the application layer is completely shielded from the intranet except through Nginx.
nexus-app network contains Nginx, FastAPI, ARQ worker, Keycloak, and Vault. Nginx forwards validated requests to FastAPI over this network. FastAPI communicates with Keycloak for JWKS key retrieval and with Vault for dynamic credentials and certificate requests.
nexus-ai network contains FastAPI, ARQ worker, Ollama Instance 1, Ollama Instance 2, Ollama Instance 3, DeBERTa NLI service, and BGE Embedding service. The inference services are reachable only from FastAPI and the ARQ worker. They are not on the data network — inference models cannot directly query databases.
nexus-data network contains FastAPI, ARQ worker, Qdrant, OpenSearch, PgBouncer (proxying PostgreSQL), Redis Instance 1, Redis Instance 2, and Vault. Data stores are reachable only from the application and task worker. They are not on the AI network — inference models cannot access databases directly.
nexus-obs network contains all services (every service exposes a Prometheus metrics endpoint on this network) plus Prometheus and Grafana. This is a read-only network from the services' perspective — they expose metrics, Prometheus scrapes them, Grafana reads from Prometheus. No service receives inbound connections from Grafana or Prometheus except through the metrics endpoint.
The security property of this topology: even if an attacker gained code execution inside the DeBERTa NLI container (which is on the nexus-ai network), they would have no network route to Qdrant, PostgreSQL, or Redis (which are on the nexus-data network). The inference containers are completely isolated from the data stores.

PART 2 — Orchestration, Query Intelligence, and Vision

Section 3 — Zone B: Orchestration, Pipeline and State Management
Overview
Zone B is the operational brain of AEGIS. It coordinates every step of request processing from the moment a validated request arrives from Zone A to the moment the final answer is delivered to the employee's browser. It manages the memory of ongoing conversations. It dispatches background work that should not block the main response path. It monitors the health of downstream services and implements graceful degradation when any service fails.
Zone B has four components: the FastAPI application server, the ARQ background task worker, the Conversation State Machine, and the Circuit Breaker Registry. These four work together as a unified orchestration layer — they are not independent services but rather tightly coordinated modules that share the same process space or communicate through Redis.
FastAPI — The Async Application Server
FastAPI is the Python web framework that implements all of AEGIS's pipeline logic. It was chosen because it is built on Python's asyncio library (making it ideal for I/O-bound workloads where requests spend most of their time waiting for external services), because it provides native WebSocket support for streaming response delivery, and because it is straightforward to understand and maintain for a small engineering team.
The Async Event Loop Model
FastAPI runs on top of uvicorn, an ASGI server. For AEGIS, FastAPI is configured to run 2 uvicorn workers. Each worker is an independent Python process with its own asyncio event loop. Having 2 workers means that even if one worker's event loop is occupied with a CPU-bound operation (however briefly), the other worker's event loop can continue processing requests from other employees.
Within each worker, the asyncio event loop handles concurrent requests through cooperative multitasking rather than threads. When a request is waiting for a Qdrant search result, the event loop switches to processing another request. When that request is waiting for a Redis read, the event loop switches again. At any given moment the event loop may be managing 20 or more in-flight requests simultaneously, each paused at an await point. This is why AEGIS can serve multiple employees concurrently without spawning a thread per employee — the asyncio model is specifically efficient for I/O-bound workloads.
Session state is stored in Redis rather than in FastAPI's process memory, so 2 workers sharing Redis for state work correctly. A request from employee A that hits worker 1 can be followed by a request from the same employee that hits worker 2, and the session state is seamlessly shared through Redis.
The Request Lifecycle Through FastAPI
When a chat request arrives from Nginx at FastAPI, it passes through a middleware stack in a defined order before reaching the route handler.
The authentication middleware runs first. It extracts the JWT from the HttpOnly cookie, verifies the signature using Keycloak's cached public key, checks the exp claim, performs the Redis SISMEMBER check for the jti against the revocation set, and extracts the user ID and role. Any failure here returns 401 immediately.
The Input Governance middleware runs second. Schema validation, file type check (for upload requests), and SAP-injection pattern scan. Any failure returns 400 immediately.
The rate limit middleware runs third. Redis INCRBY on the rate limit counter key, check against the 60/minute threshold. Any failure returns 429 immediately.
The request context middleware runs fourth. A trace ID is generated as a UUID4 and attached to the request context. This trace ID will flow through every subsequent operation for this request — it is the identifier used to correlate all log entries and metrics for this single employee interaction across all components.
Only after all four middleware stages pass does the request reach the route handler function, which begins the business logic pipeline.
The Chat Handler — Entry Point to the AI Pipeline
The chat handler is the route handler for POST /api/chat requests. It performs the following sequence of operations, all using Python await syntax to yield to the event loop during I/O waits.
It loads the employee's session state from Redis Instance 1 using the session ID from the JWT's sub claim. If no session exists (first message), it initialises a new session state object. If a session exists, it updates the TTL to reset the 2-hour inactivity timer.
It checks whether a DiagnosticObject is ready for this session (indicating that a previous screenshot upload has been processed by the ARQ vision worker). If diagnostic_object_ready is true in the session state, it fetches the DiagnosticObject from Redis and stores it in the request context for use by the retrieval and generation stages.
It invokes the Query Intelligence Layer with the raw message text and session state.
It checks the semantic cache using the enriched query embedding.
It checks the Known Patterns Registry.
It invokes the Adaptive Retrieval Engine with the enriched query and the retrieval mode determined by the Query Intelligence Layer.
It constructs the prompt.
It opens the WebSocket response channel and begins streaming generation while simultaneously running Tier 1 and Tier 2 validation.
It finalises validation after generation completes, appends the confidence badge and attribution panel, and ensures the Tier 3 LLM judge runs post-generation if applicable.
It queues ARQ background tasks: audit log write, semantic cache population if high confidence, feedback event tracking.
It updates the session state in Redis with the new conversation turn.
ARQ — The Persistent Background Task System
ARQ is the background task processing system for AEGIS. It handles all operations that should not block the main response path but must persist reliably across server restarts. ARQ runs as a separate lightweight Python process — the ARQ worker — that shares the same Docker network as FastAPI but runs independently.
Why ARQ Over Raw asyncio and Over Celery
A raw asyncio.Queue in FastAPI's memory would solve the "don't block the main path" requirement but fails the persistence requirement. If the server restarts during vision processing — which takes 30-60 seconds and is therefore highly likely to span a server restart — the in-memory queue is gone. The vision task disappears with no error, no retry, and no record. The employee's screenshot is silently ignored.
Celery solves persistence but introduces a broker process (RabbitMQ or Redis configured as Celery broker), a worker pool process, a beat scheduler for periodic tasks, and Flower for monitoring — four additional operational concerns for a team that is already managing fifteen services. Celery's async support is retrofitted rather than native, and its configuration surface is large enough to produce subtle bugs even for experienced teams.
ARQ uses Redis Instance 2 as its backing store natively. Tasks are serialised as JSON and pushed to Redis lists before execution. If the ARQ worker crashes mid-task, the task remains in Redis and is retried when the worker recovers. ARQ is fully async-native (it integrates with Python asyncio correctly), its API surface is small, and it requires no additional processes beyond the Redis instance already in the architecture. Configuring a new task type requires adding one function with a decorator — nothing more.
ARQ Task Types
Vision processing is the most important ARQ task. When an employee uploads a screenshot alongside a chat message, FastAPI saves the image bytes to the temporary filesystem at /tmp/aegis_uploads/{session_id}_{timestamp}.{ext} and queues an ARQ vision task containing only the file path and session ID — not the image bytes themselves. The ARQ worker reads the file from disk, calls Ollama Instance 3's vision API with the image, waits for the DiagnosticObject response (30-60 seconds on CPU), stores the DiagnosticObject as a JSON string in Redis Instance 1 under the key diagnostic:{session_id} with a 10-minute TTL, sets the diagnostic_object_ready flag to true in the session state hash, and publishes a vision_complete:{session_id} message to a Redis Pub/Sub channel. The FastAPI WebSocket handler is subscribed to this channel and receives the message, triggering the proactive refined response delivery described in Section 4. The ARQ worker then deletes the temporary file from the filesystem. If the Ollama vision service is unavailable, the task retries up to 3 times with 30-second delays between attempts before moving to the dead letter queue.
The reason file paths rather than image bytes are passed to ARQ is memory efficiency. A typical SAP screenshot at moderate quality is 500KB to 3MB. Base64-encoded for JSON serialisation, a 2MB image becomes 2.7MB stored in Redis Instance 2. If five employees simultaneously upload screenshots, Redis Instance 2 receives 13.5MB of task data at once. Storing the file on disk and passing the path uses negligible Redis memory.
Audit log writes are ARQ tasks to prevent audit logging from adding database latency to the request path. After every request completes — whether success or failure — FastAPI queues an ARQ audit task with the complete audit record fields. ARQ writes this to the audit_log PostgreSQL table. If the ARQ worker is temporarily unavailable, audit tasks queue up in Redis Instance 2 (which has AOF persistence) and are processed when the worker recovers. No audit records are lost.
Feedback diagnosis is an ARQ task triggered when an employee submits thumbs-down feedback. The task re-runs retrieval for the failed query, evaluates the original answer's claims against the re-retrieved chunks using DeBERTa NLI, classifies the failure as retrieval failure or generation failure, and writes to the appropriate table. The full algorithm is specified in Section 9 (Zone E, Feedback section).
Semantic cache population is an ARQ task that runs after a high-confidence answer is delivered. The task embeds the enriched query using the BGE service, stores the embedding and answer in the cache_queries Qdrant collection, and sets a 24-hour TTL reference in the payload. This is fire-and-forget — if the task fails, the answer simply is not cached, which is not a system failure.
Knowledge gap event recording is an ARQ task triggered when the CRAG assessor returns INSUFFICIENT. The task writes to the knowledge_gap_events PostgreSQL table.
Mock ticket creation is an ARQ task triggered when the session's unresolved_count reaches 3 or when CRAG returns INSUFFICIENT on a query that cannot be improved by Mode C retrieval. The task generates a ticket ID in the format TKT-{UTC-date}-{UUID4-first-8-chars}, creates a record in the mock_tickets PostgreSQL table, and publishes a WebSocket notification to the employee's session informing them that a support ticket has been created.
Nightly cache cleanup is a scheduled ARQ task that runs at 2am. It queries the cache_queries Qdrant collection for all points with created_at older than 24 hours and deletes them. This implements TTL management for the cache collection since Qdrant does not have native TTL per point.
ARQ Retry and Dead Letter Configuration
Each task type has its own retry policy configured in the ARQ task decorator. Vision tasks retry 3 times with 30-second delays. Audit log tasks retry 5 times with 10-second delays. Feedback diagnosis tasks retry 2 times with 60-second delays. Semantic cache population tasks do not retry — cache miss is acceptable. After exhausting retries, failed tasks are moved to a arq:dead_letter:{task_type} Redis list. The ARQ dead letter queue size is monitored by Prometheus and appears on the Grafana health dashboard — a growing dead letter queue is a signal for investigation.
Conversation State Machine
The Conversation State Machine manages the memory of every active employee conversation. It is not a separate service — it is a structured module within FastAPI that reads and writes a session state object stored as a Redis hash in Redis Instance 1.
The Session State Object — Complete Field Specification
For every active employee session, Redis Instance 1 stores a hash at the key session:{session_id}. The hash contains the following fields, each serving a specific purpose in the pipeline.
user_id_hash — the SHA-256 hash of the employee's JWT sub claim. Used for audit logging without storing raw user identity in session state.
created_at — ISO timestamp of when the session was initiated. Used to calculate session age for analytics.
conversation_history — a JSON-serialised array of the last 3 conversation turns. Each turn is a compact object containing: query_summary (the employee's message, truncated to 200 characters), answer_summary (the system's response, truncated to 300 characters), classification (the query type: ERROR_RESOLUTION/PROCESS/CONFIG/SIMPLE_FACT), confidence_badge (green/amber/none), retrieved_doc_ids (array of document IDs surfaced). Older turns are dropped as new ones are added. The deliberately compressed representation keeps the session hash small and the prompt context manageable.
active_retrieval_mode — the character A, B, or C indicating which retrieval mode was used in the most recent turn. Follow-up questions default to continuing at the same depth, so a complex query that activated Mode C keeps Mode C active for subsequent follow-ups on the same topic.
last_entities — a JSON-serialised array of SAP entities extracted from the most recent employee message. Format: [{"type": "error_code", "value": "VL150"}, {"type": "tcode", "value": "VL01N"}]. Used by the Context Resolver in the Query Intelligence Layer — if the employee's next message contains reference language ("that error", "same issue"), the resolver substitutes the most relevant last_entities values.
last_document_ids — a JSON-serialised array of document IDs that were included in the retrieved context of the most recent response. Used to inject a diversity preference in the next retrieval call, preventing the same chunks from dominating consecutive turns.
model_tier_last — the integer 1, 2, or 3 indicating which reasoning tier was used in the most recent generation. Follow-up questions default to at least this tier, ensuring that a complex topic that warranted Tier 3 does not suddenly receive a Tier 1 response on the follow-up.
confidence_history — a JSON-serialised array of the last 5 ValidationScore float values. If confidence has been declining across consecutive turns (each score lower than the previous), this signals that the knowledge base may not adequately cover the current topic.
unresolved_count — an integer starting at 0 and incrementing each time a turn results in either INSUFFICIENT CRAG assessment or negative feedback signal within the same session. When this count reaches 3, the pipeline automatically appends an escalation suggestion to the next response.
intent_label — a string label describing the apparent purpose of the current conversation. Format: {classification}:{primary_entity} — for example ERROR_RESOLUTION:VL150 or PROCEDURE:YDSA or CONFIG:WITHHOLDING_TAX. Generated deterministically at the end of each turn from the query classification and the primary extracted entity. If no entity was extracted, the label is the classification alone. This requires no model call — it is a simple string concatenation. The intent label is used to bias retrieval toward the relevant Qdrant collection on follow-up questions that do not mention the topic explicitly.
diagnostic_object_ready — a boolean, initially false. Set to true by the ARQ vision worker when it completes DiagnosticObject extraction and writes the result to Redis. Checked by the main pipeline on every turn to determine whether screenshot context is available.
last_updated_at — ISO timestamp of the most recent state update. Used for session age calculation.
The entire session hash has a 2-hour TTL in Redis, reset on every request. Redis handles expiry automatically — no background job is needed to clean up expired sessions. When a session expires, all session state is gone and the employee starts a fresh conversation the next time they open AEGIS.
State Transitions
At the start of every request, the session state is loaded with a single Redis HGETALL command (under 1ms). The loaded state informs every decision in the pipeline — retrieval mode, model tier, context resolver, intent label routing.
At the end of every request (after the response is delivered), the session state is updated with a single Redis HMSET command atomically updating all modified fields. The update writes: the new conversation turn appended to conversation_history (with oldest turn removed if length exceeds 3), the retrieval mode used, the entities extracted, the document IDs retrieved, the model tier used, the new ValidationScore appended to confidence_history, and the updated intent_label. The TTL is reset to 2 hours.
Circuit Breaker Registry
The Circuit Breaker Registry maintains one circuit breaker per external service. Circuit breakers prevent a failing downstream service from causing the entire pipeline to cascade — when a service is consistently failing, the circuit opens and requests fail immediately rather than hanging until timeout.
How Each Circuit Breaker Works
Each circuit breaker monitors calls to its service using a rolling window of the last 10 calls. If the failure rate within the window exceeds 50% (5 or more of the last 10 calls failed or timed out), the circuit moves to Open state. In Open state, any call to that service is immediately rejected with a fallback response — no actual call is attempted. The circuit stays Open for a 30-second cooldown period. After cooldown, it moves to Half-Open state, where the next call is allowed through as a test. If the test call succeeds, the circuit resets to Closed (normal operation). If the test call fails, the circuit returns to Open for another 30-second cooldown.
Fallback Paths by Service
When the Qdrant circuit breaker is Open, AEGIS falls back to OpenSearch-only BM25 retrieval. Vector semantic search is unavailable but keyword search continues. Answer quality degrades moderately — queries without exact term matches produce weaker results.
When the OpenSearch circuit breaker is Open, AEGIS falls back to Qdrant-only dense retrieval. Keyword search is unavailable but semantic search continues. Queries with exact error codes or T-codes may be slightly less precise without BM25 entity boosting.
When both Qdrant and OpenSearch circuit breakers are Open simultaneously, AEGIS uses the following three-stage fallback chain. First, the Known Patterns Registry in PostgreSQL is checked — if the query contains a registered entity, the pre-linked document is retrieved from the PostgreSQL parent record and used as context. Second, if no registry match exists, the semantic cache in Redis Instance 1 is checked — the query is embedded by the BGE service, compared against cached query embeddings with the standard 0.88 threshold, and if a valid cached answer exists with good freshness, it is returned with a note that it is a cached response. Third, if neither registry nor cache provides an answer, a service degradation message is returned: "The document search services are temporarily unavailable. Please try again in a few minutes or contact IT support directly."
When a model endpoint circuit breaker is Open, generation is unavailable. AEGIS checks the semantic cache first and returns a cached answer if available. If no cache entry exists, a degradation message with an estimated recovery time is returned.
When the Vault circuit breaker is Open, existing dynamic credentials continue to function until they expire. A critical alert is triggered in the observability system because prolonged Vault unavailability will eventually cause all services to lose their credentials as TTLs expire. This requires immediate operational attention.

Section 4 — Zone C-1: Query Intelligence Layer and Vision Service
Overview
The first two components of Zone C handle the two distinct input types that AEGIS receives: text queries processed by the Query Intelligence Layer, and screenshots processed by the Vision Service. These components run as early as possible in the pipeline — before retrieval, before generation — because their outputs significantly shape what everything downstream does. Understanding the query precisely and extracting structured information from screenshots are foundational to answer quality.
The Query Intelligence Layer
The Query Intelligence Layer (QIL) is a synchronous module within FastAPI that transforms the raw employee message into an enriched, structured query object that the Adaptive Retrieval Engine can work with far more effectively than the raw text. It runs entirely in-process — no network calls, no model inference — and completes in under 10 milliseconds.
The QIL is the answer to a fundamental problem in enterprise RAG: employees do not speak the same language as the documentation. An employee types "my delivery is blocked and stock is showing" when the documentation is indexed under "VL150 material availability error outbound delivery creation VL01N". Without the QIL, the embedding model must bridge this gap entirely through semantic similarity — which it does imperfectly. With the QIL, explicit entities are extracted, context is resolved, and technical vocabulary is expanded before any search begins.
Stage 1 — Entity Extraction
Entity extraction runs compiled regular expression patterns against the raw message text. The patterns are designed specifically for the SAP domain based on the patterns observed in Sona Comstar's actual SAP documentation.
The error code pattern matches sequences of one to four capital letters followed by two to six digits that appear at word boundaries — patterns like VL150, F5201, M7073, ZFI023, MARA01. The pattern includes context requirements: the matched string must be preceded or followed by a word boundary and must not be a substring of a longer alphanumeric sequence. This prevents false matches on things like "F5201GHJ" or the letter-digit sequences that appear in material numbers.
The T-code pattern matches two to five capital letters followed by one to four digits, optionally followed by a single capital letter — patterns like FB01, MIGO, VL01N, ME21N, MM02, FTXP, OB52, VA31, VF01. The same word boundary requirements apply.
The SAP document number pattern matches exactly 10-digit numeric strings surrounded by word boundaries.
The module keyword pattern is an exact match lookup against the set {FI, MM, SD, HR, PP, CO, BASIS} — these must appear as complete words, not as substrings.
Extracted entities are stored in a structured list: [{"type": "error_code", "value": "VL150"}, {"type": "tcode", "value": "VL01N"}, {"type": "module", "value": "SD"}]. This list is used by three downstream components: the Known Patterns Registry lookup (for exact entity matching), the Qdrant retrieval call (for payload filter construction), the OpenSearch BM25 call (for entity boosting), and the session state update (as last_entities for context resolution in future turns).
Stage 2 — Context Resolution
Context resolution handles follow-up questions that reference earlier parts of the conversation without naming the topic explicitly. Without context resolution, an employee's second message "what if the stock is in quality inspection?" is processed as a standalone ambiguous query — the retrieval engine has no entity to anchor on and may return completely irrelevant results.
The resolver reads last_entities from the loaded session state. It scans the current message for reference signals — a defined list of phrases and patterns that indicate the current message continues an existing topic rather than initiating a new one. The reference signal list includes: "what if", "what about", "that error", "the same issue", "it still", "this problem", "does that also", "after that", "then what", "what happens when", "how about when", "what else", "same thing". These are patterns observed in how manufacturing employees actually phrase follow-up questions in SAP support contexts.
When a reference signal is detected and last_entities is non-empty, the resolver selects the most relevant entity from last_entities. Entity priority: error_code first, then tcode, then module, then document_number. The selected entity is appended to the current message in a structured way: the original message remains intact, and [context: continuing topic {entity_type}:{entity_value}] is appended as a structured tag. This tag is visible to the retrieval engine (which uses it for filtering) but is stripped before the message reaches the generation model's visible context.
When no reference signals are detected, the message is treated as a new topic and last_entities from the previous turn is not used. This prevents false context injection when an employee genuinely changes topic.
Stage 3 — Synonym Expansion
The synonym map is a static JSON dictionary stored in the AEGIS application configuration and managed through the Admin Portal's Knowledge Management interface. It maps common employee phrasings (lowercased, trimmed) to SAP-aligned expansion terms.
The expansion lookup is a simple dictionary key check: the message text is lowercased and checked against each key in the synonym map. For multi-word keys, the check uses substring matching. If a match is found, the expansion terms are appended to the query text. The original message is preserved — expansion adds terms, it does not replace. This means the retrieval engine sees both the employee's natural phrasing and the technical SAP vocabulary.
Example: "my delivery is stuck and shows no stock" matches the synonym entry "delivery blocked" → "outbound delivery creation error VL01N VL150 material available stock". After expansion, the query contains both the employee's description and the relevant SAP terms. If the entity extractor had not found VL150 in the original text, synonym expansion may have added it, allowing the entity boosting in BM25 to fire.
The synonym map starts with approximately 20-25 entries covering the most common phrasings observed in the existing SAP documentation analysis. New entries are added by the IT admin through the Admin Portal when the knowledge gap dashboard reveals that specific phrasings are repeatedly producing INSUFFICIENT results.
Stage 4 — Query Complexity Classification and Retrieval Mode Assignment
The complexity classifier determines which retrieval mode to activate. It evaluates the enriched query against a set of deterministic signals in a specific priority order.
Mode A is assigned when the entity extraction finds at least one entity AND the Known Patterns Registry lookup returns a match for that entity. Mode A means the system knows exactly which document covers this query. The registry check runs as part of this classification — it is a single fast PostgreSQL query and its result is available within 2-3 milliseconds.
Mode B is assigned when entity extraction found entities but the registry returned no match, OR when the message is straightforward (single topic, single intent, under 150 characters, no conditional language). Mode B covers the majority of queries.
Mode C is assigned when any of the following complexity signals are present: multiple entities from different modules in the same message (suggesting a cross-module issue), conditional or temporal phrasing ("when X happens", "after Y", "before Z", "because of W"), question length exceeding 200 characters (indicating the employee has provided extensive context suggesting complexity), or the presence of multiple distinct question marks (suggesting a multi-part question).
The assigned mode is stored in the enriched query object and also updates the active_retrieval_mode in the session state. The retrieval mode assignment drives which stages of the Adaptive Retrieval Engine activate.
The Enriched Query Object
The output of the QIL is a structured Python object containing: the original raw message, the enriched message text (with synonym expansions appended), the extracted entity list, the context-resolved entity (if a reference signal was found), the assigned retrieval mode (A, B, or C), the query classification (ERROR_RESOLUTION/PROCESS/CONFIG/SIMPLE_FACT derived from entity types and message phrasing), and the registry result if Mode A was assigned (containing the linked document ID and registry notes).
This object is passed to the Adaptive Retrieval Engine and is also used to construct the six-section prompt for the Reasoning Service.
Semantic Cache Check
Between the QIL and the Retrieval Engine, the system checks whether a semantically equivalent query has been answered recently and the answer is still valid. This check uses the BGE Embedding Service and the cache_queries Qdrant collection.
The enriched query text (with expansions) is sent to the BGE Embedding Service, which returns a 768-dimensional vector. This vector is used to search the cache_queries Qdrant collection for similar previous queries. The search uses a cosine similarity threshold of 0.88 — a value calibrated to distinguish genuine semantic equivalence from superficial similarity. Below 0.88, queries may be on the same topic but different enough that the same answer would be imprecise.
If a cache hit is found (similarity ≥ 0.88), two freshness validations are performed before returning the cached answer. First, the document IDs stored in the cache entry are checked against the documents_registry PostgreSQL table — if any cited document has been updated or deprecated since the cache entry was created, the cache entry is stale and is discarded (triggering normal retrieval). Second, the cache entry's created_at timestamp is checked — entries older than 24 hours are discarded. If both freshness checks pass, the cached answer is returned through the WebSocket with a subtle "cached response" indicator in the metadata (not visible to the employee in the main answer text, but recorded in the audit log and available to the IT admin).
A valid cache hit means the employee receives their answer in approximately 300 milliseconds rather than 50-90 seconds — the entire retrieval, generation, and validation pipeline is bypassed.
If no cache hit exists, the pipeline proceeds to the Adaptive Retrieval Engine.
Vision Service
The Vision Service processes screenshots of SAP screens uploaded by employees. It extracts structured information from the screenshot that makes every subsequent pipeline stage more specific and accurate — enabling AEGIS to reference the exact material number, plant code, error code, and field values from the employee's actual SAP screen rather than speaking generically about the error type.
Why Screenshots Matter for Answer Quality
Consider an employee who uploads a screenshot showing the VL150 error popup for material CPPO3S 28A022DA at plant 1000 with batch 0000287228. Without vision processing, AEGIS produces a general answer about VL150 causes and resolution steps. With vision processing, AEGIS produces an answer that says specifically: "For material CPPO3S 28A022DA at plant 1000, the unrestricted stock shows 200 EA but the safety stock configured in MM02's MRP 2 tab for this material and plant is 300 EA, leaving 0 EA available for delivery." The specificity of the second answer is qualitatively better and immediately actionable.
The Vision Model
The vision model is Qwen2.5-VL-7B-Instruct Q4_K_M running on Ollama Instance 3. Ollama Instance 3 is configured with OLLAMA_KEEP_ALIVE=-1 to keep the model loaded in RAM permanently. Without this configuration, Ollama unloads the vision model after 5 minutes of inactivity. If an employee uploads a screenshot after the model has been unloaded, the next request triggers a 3-5 minute model reload. The ARQ vision task would appear to hang with no progress or error message. Setting KEEP_ALIVE=-1 ensures the model is always ready.
The Async Processing Flow — Phase 1: Upload to Queue
When an employee submits a message with a screenshot attachment, FastAPI's file upload handler receives the image file. FastAPI immediately saves the image bytes to the temporary filesystem at /tmp/aegis_uploads/{session_id}_{utc_timestamp_ms}.{ext}. The directory /tmp/aegis_uploads/ is created by the container entrypoint if it does not exist. Only the file path (a short string) and the session ID are passed to the ARQ vision task — not the image bytes. FastAPI then queues the ARQ vision task and continues with the main pipeline using only the text portion of the employee's message.
The Async Processing Flow — Phase 2: Vision Worker Processing
The ARQ worker process picks up the vision task from Redis Instance 2. It reads the image from the filesystem at the stored path. It calls Ollama Instance 3's API with the image, sending a structured prompt that instructs the vision model to examine the SAP screen and extract specific categories of information.
The vision prompt is carefully constructed to guide Qwen2.5-VL-7B to focus on the structured elements of SAP screens: the screen title (which indicates the transaction), the error popup content if present, the field labels and values in the main screen area, the status bar content, and any highlighted or red-coloured elements (which SAP uses to indicate errors or mandatory fields).
The DiagnosticObject
The vision model's output is parsed into a structured DiagnosticObject with the following defined fields:
error_code — the SAP message code if visible (e.g., VL150). Null if no error code is shown.
error_message_text — the complete error message text as it appears in the SAP popup or status bar. Copied verbatim, not paraphrased.
transaction_code — the T-code identified from the screen title. SAP screen titles follow the pattern "[Transaction description]: [Screen name]", which the vision model can parse reliably.
screen_title — the full title bar text of the SAP screen.
material_number — any material number visible in the screen content. SAP material numbers follow specific formats that the vision model is prompted to identify.
plant_code — any plant code visible in the screen, typically a 4-digit number like 1000, 4200, or 9000.
document_number — any SAP document number visible, typically a 10-digit number.
batch_number — any batch identifier visible.
field_values — a list of key-value pairs for important named fields visible on screen: [{"field": "Safety Stock", "value": "300"}, {"field": "Unrestricted", "value": "200 EA"}].
visible_quantities — any quantity values visible with their labels: [{"label": "Open qty", "value": "250 EA"}, {"label": "Available", "value": "0 EA"}].
Fields that are not visible in the screenshot are set to null. The vision model is explicitly prompted to use null for invisible fields rather than guessing or inferring.
The Async Processing Flow — Phase 3: DiagnosticObject Storage and Proactive Push
After extracting the DiagnosticObject, the ARQ worker serialises it as JSON and stores it in Redis Instance 1 under the key diagnostic:{session_id} with a 10-minute TTL. This TTL ensures the DiagnosticObject does not persist indefinitely if the employee never sends a follow-up.
The worker then sets the diagnostic_object_ready flag to true in the session state hash in Redis Instance 1.
The worker then publishes a vision_complete:{session_id} message to Redis Pub/Sub.
The FastAPI WebSocket handler for the employee's session subscribes to this Redis Pub/Sub channel when it opens the WebSocket connection. When the vision_complete:{session_id} message arrives, the WebSocket handler is still open (it remains open after the initial text answer is delivered, listening for both streaming tokens and vision completion signals). Upon receiving the vision completion signal, the handler automatically triggers a second response generation cycle using the original query combined with the DiagnosticObject context. The employee receives a follow-up message in their chat: "Your screenshot has been processed. Here is a more specific answer based on your exact SAP screen: [refined answer]."
This means the employee does not need to send a follow-up message to benefit from their screenshot. The system proactively delivers the enhanced answer when the screenshot processing completes, regardless of what the employee is doing.
WebSocket Connection Lifecycle
A critical implementation requirement for the proactive vision push to work: the WebSocket connection must remain open after the initial text answer is delivered. It must not be closed as soon as the response stream ends. The WebSocket handler runs a persistent listening loop after initial answer delivery, waiting for either a new message from the employee or a vision completion signal from Redis Pub/Sub. The connection is closed only when the employee closes the browser tab, the session TTL expires, or a WebSocket timeout occurs (configured at 180 seconds of total inactivity).

PART 3 — Retrieval, Reasoning, and Validation

Section 5 — Zone C-2: The Adaptive Retrieval Engine
Overview
The Adaptive Retrieval Engine is the most technically complex component in AEGIS and the one that most directly determines answer quality. Its job is to find the specific documentation chunks most relevant to the employee's query from the AEGIS knowledge base, with a precision and recall appropriate to the query's complexity.
The engine implements a three-mode adaptive pipeline. Simple queries with registry matches receive fast, targeted retrieval. Standard queries receive full hybrid dual-search with knowledge graph expansion and intelligent quality assessment. Complex multi-aspect queries receive deep multi-query retrieval with diversity-boosted merging. The mode is determined by the Query Intelligence Layer before the retrieval engine runs.
The engine has eight sequential stages, though not all stages activate for every query. Understanding each stage and when it activates is essential for understanding why AEGIS retrieves better than a simple single-vector-search system.
Stage 1 — Known Patterns Registry Lookup (Mode A Trigger)
The Known Patterns Registry is a PostgreSQL table that stores Sona Comstar-specific known patterns — entities that AEGIS can immediately map to specific documentation without needing to search. The lookup is a fast parameterised PostgreSQL query with an index on the pattern_string column, completing in under 2 milliseconds.
The Registry Schema
The known_patterns_registry table contains: pattern_string (the exact entity string to match — "VL150", "YDSA", "9000", "G5"), pattern_type (enum: error_code/order_type/plant_code/tax_code/pricing_procedure/transaction), linked_document_id (the document that most directly covers this pattern — "SD-ERR-001"), linked_chunk_type (which chunk type to prioritise in retrieval — for error codes, cause_resolution chunks are most valuable), registry_notes (context injected into the generation prompt when this pattern matches — e.g., "Plant 9000 is the Sona Comstar GIT Warehouse. Storage location field Y013 is mandatory for this plant. All schedular agreements at plant 9000 require a storage location."), status (enum: draft/approved/deprecated), approved_by, created_at, approved_at.
How Mode A Uses the Registry
When Mode A is assigned by the QIL (entity found AND registry match exists), the registry result provides the linked document ID and registry notes. The retrieval engine uses the linked document ID to construct a hard Qdrant payload filter: document_id == linked_document_id. The Qdrant HNSW search operates only within this filter — it traverses only the four to eight chunks belonging to the matched document rather than all chunks in the collection.
Within this filtered search, both the content vector and identity vector searches run. All chunks from the matched document are returned (typically 3-8 chunks for a complete error guide or procedure). The registry notes are extracted and stored for injection into the prompt's context section.
Mode A then skips Stages 2, 3, 4, 5, and 6 (OpenSearch BM25, Knowledge Graph, RRF fusion, CRAG, and Mode C multi-query) because the retrieval is already targeted to a specific known document. It proceeds directly to Stage 7 (cross-encoder reranking) to order the document's chunks by relevance to the specific question within the known document.
This fast path produces excellent results for known error codes and known Sona Comstar-specific configurations because the registry provides a guaranteed link between the entity and its documentation — no similarity search required.
Stage 2 — Qdrant Dense Vector Search
For Mode B and Mode C queries, the Adaptive Retrieval Engine performs dense vector search in the Qdrant collection corresponding to the query classification.
Collection Selection
The QIL's query classification drives collection selection: ERROR_RESOLUTION queries search meridian_errors, PROCESS queries search meridian_procedures, CONFIG queries search meridian_configs. SIMPLE_FACT queries search all three collections with results merged before RRF fusion.
This collection separation is a fundamental quality improvement over a single unified collection. If all content types shared one collection, a query about "VL150 delivery error" would compete semantically against procedure chunks about scheduling agreements, configuration entries about withholding tax, and other error guides about completely different errors — all in the same vector space. The HNSW search would find the nearest neighbors across this mixed semantic landscape, producing noise in the results. Separate collections mean the HNSW search operates only within the semantically coherent space of documents of the same type, producing higher-precision nearest neighbors.
Two-Vector Search
Each chunk in Qdrant has two named vectors: the content vector (768-dimensional BGE-base embedding of the full chunk text) and the identity vector (768-dimensional BGE-base embedding of the document ID, error code or configuration name, and module as a string).
The query is embedded twice by the BGE Embedding Service — once to produce a query content vector and once to produce a query identity vector. The content vector captures the semantic meaning of the employee's question. The identity vector captures what the employee is asking about at the concept level.
Both vector searches run simultaneously as two named vector queries to Qdrant. Qdrant returns two ranked lists: one ordered by content vector similarity and one ordered by identity vector similarity. These two lists are pre-merged within the Qdrant response before going to the main RRF fusion stage — the pre-merge uses RRF within the collection (content rank + identity rank) to produce a single unified per-collection ranking. This means each collection search contributes one ranked list to the main RRF fusion rather than two, keeping the fusion arithmetic clean.
HNSW Configuration
The Qdrant collections are configured with HNSW parameters tuned for quality: m=32 (each node connects to 32 neighbors during index construction — higher than the default 16, improving recall at the cost of index size), ef_construction=200 (200 candidates evaluated during index building — higher values produce better-quality graphs), ef=128 (128 candidates examined during query time — higher values improve recall at the cost of search latency).
Scalar quantization is applied to reduce memory footprint. The 768-dimensional BGE-base vectors, which are 32-bit floats (3072 bytes per vector), are quantized to 8-bit integers (768 bytes per vector) — a 4x reduction in memory. The quality loss from scalar quantization is less than 2% for BGE-base vectors because the vector components are well-distributed and the quantization error is small relative to inter-vector distances.
Payload Filters for Module Scoping
When the entity extraction found a module keyword (FI, MM, SD, HR, PP, CO), the Qdrant search includes a payload filter: module == extracted_module. This restricts the HNSW traversal to chunks belonging to documents of the detected module, improving precision significantly. A query that explicitly mentions "FI" should not retrieve MM procedures with the same relevance as FI configurations.
When no module was extracted, no module filter is applied and the search covers all chunks in the collection.
Result Count
Qdrant returns 10 candidate results per collection (or 10 per collection for SIMPLE_FACT queries searching all three). These 10 results per collection, with their similarity scores, proceed to Stage 4 (OpenSearch BM25 search proceeds in parallel to Stage 2, not after it).
Stage 3 — OpenSearch BM25 Search (Runs in Parallel With Stage 2)
OpenSearch BM25 search runs concurrently with the Qdrant vector search using asyncio.gather() — both searches are dispatched simultaneously and their results are collected when both complete. This parallel execution means the combined latency of Stages 2 and 3 is approximately equal to the slower of the two (typically Qdrant, at 30-80ms on the demo hardware) rather than the sum of both.
The SAP-Specific Custom Analyzer
OpenSearch's custom analyzer is what differentiates AEGIS's keyword search from generic BM25. The analyzer pipeline processes text through the following filters in order when indexing and querying:
The SAP entity tokenizer is a custom token filter that applies pattern-matching to the token stream. When it identifies a token matching the SAP error code pattern (capital letters followed by digits) or the T-code pattern, it marks that token as a SAP entity and prevents subsequent filters from modifying it. This ensures "VL150" is indexed and searched as the single token "VL150" rather than being split or stemmed.
The standard tokenizer handles all other text, splitting on whitespace and punctuation.
The lowercase filter converts all non-entity tokens to lowercase.
The SAP synonym filter applies predefined expansions. This filter maps both directions: it expands employee phrasing to technical terms and also allows technical terms to match employee phrasings. The synonym file contains entries like: goods receipt, GR, MIGO => goods receipt GR MIGO material document, invoice verification, invoice, MIRO => invoice verification MIRO FI, purchase order, PO, ME21N => purchase order PO ME21N MM.
The English stop word filter removes common English words that carry no retrieval signal: the, and, is, for, to, etc.
Entity Token Boosting in Queries
Before the query text reaches OpenSearch, the QIL's extracted entities are duplicated in the query string to amplify their BM25 weight. An extracted entity appears three times in the query string sent to OpenSearch (the original occurrence plus two duplicates). BM25 scores higher when the query term appears more frequently relative to the document frequency — the triple occurrence makes entity matches score far above non-entity term matches in BM25 scoring.
For a query about "VL150 error during delivery", the OpenSearch query string becomes "VL150 VL150 VL150 error during delivery outbound delivery creation material available stock" (with entity boosting and synonym expansion). The BM25 scores for chunks containing "VL150" will dominate the result set.
BM25 Scoring
OpenSearch BM25 scores each matching document using the BM25 formula which accounts for three factors: term frequency (how often the query term appears in the document — higher is better, with diminishing returns), inverse document frequency (how rare the term is across the entire corpus — rarer terms are more discriminating, getting higher weight), and document length normalisation (shorter documents that densely match the query score higher than longer documents that mention the query terms occasionally). The combination of these three factors produces relevance scores that reflect genuine keyword match quality rather than raw frequency.
OpenSearch returns 10 candidate results with their BM25 scores.
Stage 4 — Knowledge Graph Traversal
The Knowledge Graph provides a third source of candidate documents beyond vector search and keyword search. It captures the structured cross-references between documents that were explicitly encoded in the AEGIS templates — the RELATED_ERRORS field in config documents and the COMMON_ERRORS_IN_THIS_PROCEDURE field in procedure documents.
The Knowledge Graph Structure
The Knowledge Graph is implemented as rows in the document_relationships table in PostgreSQL. Each row captures one directional relationship: from_document_id (the source document), to_document_id (the target document), relationship_type (causes_error/common_in_procedure/related_to). These rows are created automatically during document ingestion by parsing the template cross-reference fields.
The Traversal Query
After Qdrant and OpenSearch return their candidate sets, the Knowledge Graph traversal takes the union of all document IDs that appeared in either set and queries the document_relationships table:
"Select all document relationships where from_document_id or to_document_id appears in the candidate document IDs."
This returns all documents that are explicitly linked to any document that appeared in the search results. For example, if SD-PROC-001 (scheduling agreement procedure) appeared in the Qdrant results, the Knowledge Graph query returns all documents referenced in SD-PROC-001's COMMON_ERRORS_IN_THIS_PROCEDURE field — which includes SD-ERR-001 (VL150 error guide) and SD-ERR-002 (incompletion log error).
These Knowledge Graph expansion documents are added to the candidate pool with a base relevance contribution equivalent to a rank-12 position in the RRF calculation — meaningful but not dominant. They enrich the context without overriding the direct search results.
Stage 5 — RRF Fusion
RRF (Reciprocal Rank Fusion) combines the three ranked lists from Qdrant, OpenSearch, and Knowledge Graph expansion into a single unified ranking. RRF was chosen because it is robust to the fundamental problem of combining rankings from systems with incomparable scores — Qdrant scores are cosine similarities between 0 and 1, OpenSearch BM25 scores are in arbitrary units depending on corpus statistics, and Knowledge Graph contributions are categorical. RRF does not use the scores at all — it uses only the rank positions.
The RRF Formula
For every unique chunk that appears in any of the three lists, an RRF score is calculated:
RRF_score = sum over all lists of [1 / (rank_in_list + K)]
Where K is a constant set to 60 (the value validated empirically in the original RRF research paper as providing robust performance across diverse retrieval tasks). A chunk ranked first in Qdrant contributes 1/(1+60) = 0.0164 to its RRF score. A chunk ranked fifth contributes 1/(5+60) = 0.0154. A chunk that does not appear in a list contributes 0 from that list.
Why K=60 Prevents Top-Rank Dominance
Without the constant K, the formula would be 1/rank — a first-place result in one list would contribute 1.0 (infinitely more than any lower-ranked result). With K=60, the difference between rank 1 and rank 2 is small: 1/61 vs 1/62. This means a chunk that ranks consistently well across multiple lists (say, rank 3 in Qdrant, rank 4 in OpenSearch, and rank 2 in Knowledge Graph) scores higher than a chunk that ranks first in one list but does not appear at all in the others. This consensus-based ranking is exactly what retrieval quality requires — the best answer is the one that multiple independent search methods agree on, not the one that one method happens to rank first.
Result After RRF
The RRF calculation produces a unified ranked list of up to 28 unique chunks (10 from Qdrant + 10 from OpenSearch + up to 8 from Knowledge Graph expansion, deduplicated). The top 10 chunks from this unified list proceed to the CRAG quality assessment.
Stage 6 — CRAG Self-Reflection Quality Assessment
CRAG (Corrective Retrieval Augmented Generation) in AEGIS is a self-reflection mechanism where the system asks the Qwen2.5-7B model to assess whether the retrieved chunks are sufficient to answer the employee's question before generating a full answer. This prevents the Reasoning Service from generating a long, confident-sounding answer from chunks that are topically related but do not actually address the specific question.
Critical: CRAG Is Gated — It Does Not Run for Every Query
The previous NEXUS architecture ran CRAG on every query, adding 20-40 seconds of 7B model inference to every single response. On CPU at 3-6 tokens per second, this was architecturally unsustainable. AEGIS gates CRAG precisely: it runs only when retrieval confidence is genuinely uncertain.
CRAG is explicitly skipped when:

Mode A is active (registry match found) AND the cross-encoder top score after reranking exceeds 0.82. Registry-matched, well-retrieved content does not need self-reflection — the system already knows exactly which document covers this entity.
Mode B retrieval completes AND the cross-encoder top score exceeds 0.80. Strong retrieval confidence means the chunks are almost certainly relevant and the model does not need to reflect on them before generating.

CRAG runs when:

Mode B completes AND the cross-encoder top score falls between 0.55 and 0.80. Moderate retrieval confidence where self-reflection can identify whether the chunks are adequate or whether a deeper search (Mode C) should be attempted.
Mode C was already activated by the QIL complexity classifier. Complex queries always benefit from CRAG reflection to verify that the multi-query retrieval covered all aspects of the question.
Any cross-encoder top score falls below 0.55. Poor retrieval strongly suggests the knowledge base does not contain relevant information — CRAG should return INSUFFICIENT rather than allowing generation from irrelevant chunks.

In practice, Mode A registry hits (which cover the most common known errors) and Mode B queries with high cross-encoder confidence (which cover the majority of well-documented queries) both skip CRAG. CRAG fires on approximately 25-30% of queries.
The Self-Reflection Prompt
When CRAG fires, the Qwen2.5-7B model on Ollama Instance 2 receives a carefully structured prompt. The prompt presents the employee's enriched query and the top 8 chunks from the RRF result (their text content, chunk types, and document IDs). The model is asked to evaluate a specific set of questions: Do these chunks contain the information needed to completely answer this specific question? If yes, which chunks are most directly relevant? If no, what specific information is missing — and can you describe the gap precisely enough that a targeted additional search could fill it?
The model is constrained to respond in a specific JSON format with defined fields: assessment (the string "SUFFICIENT" or "INSUFFICIENT"), relevant_chunk_ids (list of the most relevant chunk IDs if SUFFICIENT), gap_description (a precise description of what is missing if INSUFFICIENT — for example, "The retrieved chunks describe the causes of VL150 but do not include resolution steps for the specific cause where reservations exceed available stock").
CRAG Pipeline Outcomes
If the assessment is SUFFICIENT, the top 8 chunks from RRF proceed to cross-encoder reranking. The relevant_chunk_ids list from CRAG's response is used to give additional weight to the identified high-relevance chunks during reranking.
If the assessment is INSUFFICIENT and Mode B was active, Mode C deep retrieval is activated. The gap_description from CRAG's response is used to construct one additional targeted sub-query. For example, if the gap description says "missing resolution steps for reservation cause", the sub-query becomes "VL150 resolution steps reservation cause MB25 stock overview". This sub-query runs through both Qdrant and OpenSearch search stages. The new results are merged into the existing RRF pool with a diversity bonus — chunks from new documents get a 15% upward score adjustment in the combined RRF to encourage diversity. CRAG self-reflection runs again on the enriched pool. If it returns SUFFICIENT this time, retrieval proceeds normally.
If CRAG returns INSUFFICIENT after Mode C enrichment, or if the cross-encoder score was below 0.55 on the initial retrieval, the INSUFFICIENT pipeline activates: the generation stage is bypassed, the INSUFFICIENT response is prepared ("I cannot find sufficient documentation in the knowledge base to answer this query reliably. This may be an undocumented scenario. Please raise a support ticket."), an ARQ task is queued to write a knowledge gap event to PostgreSQL, and the unresolved_count in the session state is incremented.
Multi-Query Generation in Mode C
When Mode C is activated (either by the QIL complexity classifier before retrieval or by CRAG after Mode B retrieval), the Qwen2.5-7B model generates 1-2 focused sub-queries targeting the specific gaps identified. For QIL-activated Mode C, the model is given the original query and asked to generate sub-queries that approach the multi-aspect question from different angles — one targeting each identified module or aspect. For CRAG-activated Mode C, the model is given the gap description and asked to generate one targeted sub-query addressing exactly the identified missing information.
Sub-query generation is a short model call (20-30 tokens of output) taking 5-10 seconds on CPU. The sub-queries are added to the retrieval pool and the complete Mode C retrieval proceeds.
Stage 7 — Cross-Encoder Reranking
After the CRAG gate, the top 8 chunks from the RRF unified list proceed to cross-encoder reranking. The cross-encoder provides a more accurate relevance judgment than the embedding similarity scores from Stage 2 or the BM25 scores from Stage 3.
Why Cross-Encoders Are More Accurate Than Embedding Models
Embedding models encode the query and document independently — the query produces one vector, the document produces another, and similarity is measured between them. This means the model never directly compares query tokens against document tokens. Relationships like "the employee asks about 'causing factor' and the document says 'root cause'" require the model to have learned that "causing factor" and "root cause" are semantically close during training, and have represented this in the embedding space. This works well at a coarse level but misses fine-grained relationships.
A cross-encoder receives the query and document concatenated into a single input and runs a single forward pass that produces a scalar relevance score. Every token in the query attends to every token in the document through the transformer attention mechanism. This means the model can directly identify that the document's "safety stock value in MRP 2 tab" is directly responsive to the query's "why does my stock show available but delivery shows zero" — a relationship that requires understanding multiple terms and their SAP-specific meaning together. Cross-encoders consistently outperform bi-encoders on relevance judgment at the cost of higher per-item inference time.
The Reranking Process
The reranker is ms-marco-MiniLM-L-12-v2 — a 33M parameter cross-encoder model trained on the MS MARCO passage retrieval dataset. It runs on CPU via the DeBERTa NLI service process (they share the same service since both are small CPU inference models). Each inference call receives the enriched query text and one chunk text, producing a relevance score between 0 and 1. Eight chunks require eight inference calls.
Each inference call takes approximately 25-40 milliseconds on CPU. Eight calls run sequentially within the service, totalling approximately 200-320 milliseconds. This is acceptable within the overall pipeline budget.
The 8 chunks are re-sorted by their cross-encoder relevance scores in descending order. The top 5 chunks after reranking become the final retrieved context.
Stage 8 — Parent Context Hydration
The final stage of the Retrieval Engine ensures the generation model always has document-level identity information available, regardless of which specific chunks ranked highest.
Why Hydration Is Necessary
The chunk that scores highest for a specific question is often a cause-resolution pair or a procedure step — not the document header chunk. The header chunk (which contains the document ID, error code, module, transactions, and verification information) may have ranked 6th or 7th because it does not directly answer the specific question asked.
Without hydration, the generation model receives the top 5 ranked chunks but may not know which document they are from, what error code they cover, when they were verified, or who verified them. It cannot produce a proper attribution panel. It may not even know the name of the configuration it is describing.
The Hydration Process
After the top 5 chunks are selected by reranking, the Retrieval Engine checks whether a header chunk (chunk_type == "header" or chunk_type == "procedure_header" or chunk_type == "config_overview") for the primary retrieved document is already among the 5 chunks. The primary retrieved document is the one that contributed the most chunks to the top 5.
If the header chunk is already in the top 5, no hydration is needed.
If the header chunk is not in the top 5 (because other chunk types ranked higher), the Retrieval Engine queries the PostgreSQL documents_registry table for the full parent record of the primary document. The parent record stores the complete document header field values as a JSON blob. This information is prepended to the prompt context as a structured document header block, clearly labelled as document identity information rather than chunk content.
This ensures that regardless of which specific chunks were most relevant to the query, the generation model always knows: which document it is reading from, what error or configuration the document covers, which module and transactions it applies to, who verified it, and when it was last verified.

Section 6 — Zone C-3: Reasoning Service
Overview
The Reasoning Service generates the employee's answer. It receives the assembled context from the Retrieval Engine — the top 5 chunks, the parent document header information, and any DiagnosticObject from screenshot processing — assembles the complete prompt, routes the prompt to the appropriate model tier, manages the streaming delivery of generated tokens to the employee's browser, and coordinates with the Validation Engine running concurrently.
The Model Gateway
The Model Gateway is the abstraction layer that shields the rest of AEGIS from the specifics of which inference backend (Ollama for demo, vLLM for production) is being used. It exposes a single generate(prompt, tier, stream) interface. Every call to a language model in AEGIS goes through this gateway.
OpenAI-Compatible API Standard
Both Ollama (demo) and vLLM (production) expose an OpenAI-compatible API — the same endpoint format, the same request/response JSON structure, the same streaming protocol used by OpenAI's API. The Model Gateway communicates with both backends using this standard format. Switching from Ollama to vLLM in production requires only a base URL change and an API key configuration update — no application code changes.
The gateway manages three operational concerns transparently. Connection pooling maintains persistent HTTP connections to each Ollama instance, eliminating connection establishment overhead on each request. Retry logic with exponential backoff retries failed model calls up to 2 times with increasing delays before propagating the failure to the circuit breaker. Streaming relay receives the server-sent event stream from Ollama's streaming endpoint and relays tokens to the Redis Pub/Sub channel for the employee's WebSocket.
Tier Selection Logic
AEGIS uses model tiers to balance answer quality against response latency on a per-query basis. In the demo configuration, all tiers use the same underlying model (Qwen2.5-32B) but differ in prompt strategy — specifically, whether chain-of-thought reasoning is included in the prompt. In production, tiers use different model sizes (7B, 14B, 72B), making the quality difference more pronounced.
Tier 1 — Standard Prompt, Direct Answer
Activates when: Mode A registry hit AND cross-encoder top score above 0.82 (CRAG was skipped), OR Mode B with cross-encoder top score above 0.80 (CRAG was skipped), AND the query classification is SIMPLE_FACT or a well-defined single-aspect question.
Prompt strategy: standard direct-answer prompt without chain-of-thought. The model is instructed to read the context and answer the question concisely and precisely. Expected response length: 100-300 tokens. Expected generation time on demo CPU with 32B model: 20-60 seconds.
Tier 2 — Enhanced Prompt, Structured Answer
Activates when: Mode B with CRAG returning SUFFICIENT, AND the query classification is ERROR_RESOLUTION or CONFIG.
Prompt strategy: structured answer prompt that instructs the model to organise its response with a brief diagnosis followed by numbered resolution steps followed by a verification check. The structure mirrors the template's cause-resolution format, ensuring the answer's information architecture aligns with how the documentation is organised. Expected response length: 200-500 tokens. Expected generation time: 40-90 seconds.
Tier 3 — Chain-of-Thought Prompt, Comprehensive Answer
Activates when: Mode C retrieval was used (complex multi-aspect query), OR the unresolved_count in the session state is 2 or above (the employee has asked about this topic before without resolution — the previous tier answers were insufficient), OR the query classification is PROCESS (procedural guidance benefits from reasoning through each step).
Prompt strategy: chain-of-thought prompt that first asks the model to reason through the problem before formulating the answer. The reasoning is internal (not shown to the employee) but guides the model to consider all relevant aspects before generating the final response. Expected response length: 400-800 tokens. Expected generation time: 60-150 seconds.
The tier selection updates the model_tier_last field in the session state so that follow-up questions default to at least this tier.
The Complete Prompt Structure
Every prompt sent to the generation model consists of six sections assembled in a defined order. Understanding this structure explains how the model is guided to produce accurate, policy-compliant, well-cited answers.
Section 1 — System Role
The system role establishes the model's identity and fundamental constraints. It specifies that the model is AEGIS, Sona Comstar's SAP helpdesk AI assistant. It establishes the primary constraint: the model answers only based on the provided documentation — it does not draw on training memory for facts about Sona Comstar's SAP configuration, procedures, or error patterns. It establishes the citation requirement: the model must indicate which document ID its information comes from. It establishes the uncertainty requirement: if the documentation does not contain enough information to answer completely, the model must say so explicitly rather than supplementing with general SAP knowledge that may not apply to Sona Comstar's specific configuration.
Section 2 — Policy Rules
The policy rules section specifies the operational constraints that override the model's default behaviour patterns. Constraint 1: do not instruct employees to execute T-codes requiring IT administrator or SAP consultant access without explicitly labelling them as requiring such access. Constraint 2: never reproduce or describe SAP credentials, passwords, system user IDs, or authorisation object details. Constraint 3: when documentation shows a specific Sona Comstar configuration value (for example, company code 1000 maps to Comstar India), use that specific value rather than generic SAP defaults. Constraint 4: when the documentation describes a step as requiring "IT admin" or "SAP consultant", reproduce that language exactly — do not rephrase it as something the employee can do themselves. Constraint 5: if a Config Snapshot staleness warning was injected into the context section, include the staleness disclaimer in the answer.
Section 3 — Retrieved Context
The retrieved context section presents the top 5 chunks in logical reading order: document header first (from parent hydration if needed), then cause-resolution chunks in cause number order, then outcome chunks. Config documents follow: overview first, then current values, then navigation. Procedure documents follow: header first, then step groups in phase order, then outcome.
Each chunk is presented with a structured label: [Document: {document_id} | Type: {chunk_type} | Verified: {last_verified_date} by {verified_by}]. This labelling allows the model to cite specific sources and allows the Validation Engine to verify citations. The generation model is instructed to cite sources in the format "(Source: {document_id})" when making specific claims.
Registry notes from the Known Patterns Registry are injected at the end of the context section, labelled as [Registry Context]. These notes contain Sona Comstar-specific contextual information that supplements the document content.
If a Config Snapshot was consulted for the query (CONFIG classification with SIMPLE_FACT aspects), the relevant snapshot rows are presented at the end of the context section labelled [Config Snapshot: last verified {date}]. If the snapshot age exceeds 35 days, a staleness note is injected: [Config Snapshot WARNING: These values were last verified {N} days ago. They may not reflect the current system state.].
Section 4 — Diagnostic Context (Conditional)
This section appears only when a DiagnosticObject is available from screenshot processing. It presents the structured fields extracted from the screenshot in a clear format: Error Code, Error Message, Transaction, Material Number, Plant Code, Batch Number, Visible Quantities, and Field Values. The model is instructed to use these specific values when making statements about the employee's situation — "for material {material_number} at plant {plant_code}" rather than generic references.
Section 5 — Conversation History
The last 3 turns from the session state's conversation_history field are presented in compressed format: Turn 1: Employee asked about {query_summary}. AEGIS answered about {answer_summary}. Confidence: {confidence_badge}. This allows the model to understand the conversational context — if the employee has already tried step 1 of a resolution and is asking about step 2, the model knows not to repeat step 1.
Section 6 — Current Query
The employee's enriched query, stripped of any QIL processing annotations (such as the context-resolution tag). This is exactly what the employee typed (with synonym expansions appended but internal annotations removed).
Token Streaming Delivery
The generation model produces tokens one at a time in a server-sent event stream from Ollama's streaming API endpoint. The Model Gateway's streaming relay function processes these events as an async iterator: for each token event received, it immediately publishes the token to the Redis Pub/Sub channel stream:{session_id}.
The FastAPI WebSocket handler is an async coroutine subscribed to this Redis Pub/Sub channel. When a token arrives on the channel, the handler immediately writes it to the WebSocket connection to the employee's browser. The employee sees text appearing token by token in the chat interface. The first tokens appear within 5-8 seconds of query submission on the demo CPU configuration, determined by the time for the model to process the full prompt and generate the first token.
This streaming architecture ensures the employee experiences a responsive, live interface despite the 32B model's relatively slow generation rate on CPU. Psychologically, text appearing progressively is far more acceptable than a blank waiting screen, even if the total time to complete the response is the same.

Section 7 — Zone C-4: Validation and Quality Assurance Engine
Overview
The Validation Engine is the quality gate between the generated answer and the employee. It implements three independent quality signals combined into a single ValidationScore. The three signals must be genuinely independent — if the system that generated the answer also assessed it, the assessment would be subject to the same biases and errors as the generation. Each validation tier uses a different mechanism: deterministic rules, a separate NLI classification model, and a separate LLM instance with a different system prompt.
The critical design improvement in AEGIS over the NEXUS design: validation runs concurrently with generation, not after it. The employee sees the response building in real time while validation runs in the background. Validation failures are handled either by mid-stream interruption (for Tier 1 hard failures) or by post-generation correction messages (for Tier 2 and Tier 3 findings).
Concurrent Validation Architecture
When the Reasoning Service begins streaming tokens from Ollama, the FastAPI handler opens two concurrent processing paths using Python's asyncio.create_task() function.
Path 1 — Token Streaming consumes the Ollama token stream and forwards each token to the Redis Pub/Sub channel for browser delivery. It also maintains a sentence buffer — accumulating tokens until it detects a sentence-ending signal (period followed by a space, question mark, exclamation mark, or newline). When a sentence is completed, the sentence text is extracted from the buffer and dispatched to Path 2.
Path 2 — Concurrent Validation receives completed sentences from Path 1 as they arrive during streaming. For each sentence: Tier 1 Output Governance checks run synchronously (under 2ms each, scanning the sentence for restricted patterns). If a Tier 1 check fires, a flag is set in the validation context and Path 1 is notified. A Tier 2 NLI validation task is dispatched as an asyncio.create_task() for the sentence's factual claims — this runs concurrently with Path 1 continuing to stream subsequent sentences. The Tier 2 tasks collect their results into a shared list.
asyncio.create_task() is the correct Python primitive here: it schedules the NLI validation coroutine to run on the event loop without blocking the current coroutine. Path 1 continues streaming tokens while multiple Tier 2 NLI tasks are pending. By the time generation completes, most sentences have already been evaluated — the total NLI latency is approximately equal to the time needed to evaluate the last few sentences rather than the sum of all sentence evaluation times.
Tier 1 — Deterministic Rule Checks (< 5ms per sentence)
Tier 1 runs synchronously for each completed sentence during streaming. It checks three categories of content.
Output Leak Detection scans the sentence text for patterns that should never appear in a public AEGIS answer: Docker internal IP addresses in the 172.x.x.x range, JWT token patterns (the distinctive three-part base64-dot format), Vault path patterns (strings beginning with /v1/), system prompt fragment patterns (exact phrases from the AEGIS system role text that would indicate the model echoed its instructions). If any match is found, the matched content is redacted from the sentence and a leak_detected flag is set in the validation context.
Scope Boundary Check verifies every document citation in the sentence. The generation model is prompted to cite sources as "(Source: {document_id})". When a citation appears in a sentence, the cited document ID is looked up in the PostgreSQL documents_registry table. If the document ID does not exist or has status "deprecated", a scope_violation flag is set. The specific hallucinated citation is recorded for use in the targeted regeneration prompt.
T-Code Policy Check extracts every T-code from the sentence (using the same regex pattern as the QIL entity extractor). For each extracted T-code, it queries the transaction_code_permissions table in PostgreSQL to determine the access level: employee-accessible or IT-admin-only. If an IT-admin-only T-code appears in a sentence that instructs the employee to execute it themselves (detected by examining the surrounding verb context: "navigate to", "open", "execute", "enter"), a policy_violation flag is set.
Tier 1 Hard Failure Response
If a leak_detected, scope_violation, or policy_violation flag is set, the validation context records the specific issue. If it is a hard failure (leaked content or scope violation), Path 1 is signalled to pause streaming. The employee's browser shows a brief message: "Correcting response..." A targeted regeneration is initiated immediately: the same six-section prompt is sent to the generation model with an additional Section 7 appended: "Your previous response contained the following issue: {specific_issue_description}. Please regenerate, correcting exactly this issue. Do not change any other part of the response." If the regenerated response also fails Tier 1, the answer is suppressed entirely and the escalation response is returned.
For T-code policy violations, the violation is recorded but streaming is not paused — the response is delivered with a correction appended post-generation: "Note: The T-code {tcode} mentioned in step {N} requires SAP consultant or IT administrator access. Please contact your SAP team to execute this step."
Tier 2 — NLI Entailment Check (background coroutine, under 80ms per sentence)
The DeBERTa-v3-large-mnli Model
The NLI model is DeBERTa-v3-large-mnli, a 304-million-parameter model trained on the Multi-Genre Natural Language Inference dataset. NLI is the task of determining whether a "hypothesis" sentence is logically supported by a "premise" sentence. In AEGIS, hypotheses are factual claims extracted from the generated answer, and premises are the retrieved documentation chunks.
DeBERTa classifies the relationship as ENTAILMENT (the premise logically supports the hypothesis), NEUTRAL (the premise is topically related but does not directly support or contradict the hypothesis), or CONTRADICTION (the premise directly contradicts the hypothesis). ENTAILMENT means the claim is grounded. NEUTRAL means the claim may be true but the documentation does not specifically confirm it. CONTRADICTION means the model generated something that contradicts its own source.
Claim Decomposition
Before NLI evaluation, each completed sentence is decomposed into atomic factual claims. A factual claim is any sentence or clause that asserts a specific fact, a required step, a configuration value, a causal relationship, or a conditional statement. "The VL150 error occurs when available stock minus safety stock is less than the delivery quantity" is a factual claim. "This is a common configuration issue" is not — it is a general observation with no specific verifiable assertion.
Multi-clause sentences are split at conjunctions and punctuation boundaries. "Open MM02, navigate to MRP 2 tab, and check the Safety Stock field" becomes three claims: "Open MM02", "navigate to MRP 2 tab from MM02", "check the Safety Stock field in the MRP 2 tab". This granularity ensures that if one step in a sequence is wrong, it is identified specifically rather than the entire sentence being flagged.
The 350-Token Windowing Fix
DeBERTa-v3-large has a maximum input sequence of 512 tokens. The NLI evaluation sends a claim as the hypothesis and a chunk as the premise in a single input: [CLS] claim [SEP] chunk [SEP]. A cause-resolution chunk containing 6 resolution steps can be 350-450 tokens alone. Adding the claim and separator tokens easily exceeds 512 tokens.
Before any chunk is used as an NLI premise, its token count is measured (using the DeBERTa tokenizer). Chunks under 350 tokens are used directly. Chunks between 350 and 800 tokens are split into overlapping windows: each window is 300 tokens long with a 75-token overlap between consecutive windows. For example, a 500-token chunk becomes two windows: tokens 1-300 and tokens 226-500 (with a 75-token overlap for context continuity). Each window is evaluated as a separate premise against the claim. The maximum ENTAILMENT score across all windows is taken as the chunk's entailment score for that claim.
For config_values chunks which can be very long (containing tables of current values for multiple company codes), the same windowing applies. Because relevant values are typically concentrated in a few consecutive lines, at least one window will capture the specific content relevant to the claim being evaluated.
Entailment Thresholds
A claim is considered grounded if its maximum ENTAILMENT score across all evaluated chunk-windows is above 0.80. The 0.80 threshold was established as the standard for factual claims about SAP procedures and error codes based on the observation that claims directly drawn from documentation score 0.85-0.95, while claims paraphrased at medium distance score 0.70-0.85, and claims not in the documentation score below 0.70.
For claims that reference Sona Comstar's specific configurations — values from the Config Snapshot, company-specific thresholds, module-specific rules — the threshold is raised to 0.90. These high-stakes specific claims must be more precisely grounded because acting on a wrong Sona Comstar-specific claim has higher consequences than acting on a wrong general SAP procedure claim.
The NLI Support Score for the full answer is: (number of grounded claims) / (total number of evaluated claims).
Specific unsupported claims are identified by their sentence text and included in the ValidationResult.unsupported_claims list. This list is used in targeted regeneration: "The following claims in your response are not supported by any of the retrieved documentation: {claim_1}, {claim_2}. Please regenerate, replacing these claims with information that is directly present in the provided context."
Tier 3 — LLM-as-Judge (post-generation, conditional)
Tier 3 runs after generation completes — it does not block the streaming response. It runs only for ERROR_RESOLUTION and PROCESS query types, where step completeness and logical ordering are meaningful quality dimensions. SIMPLE_FACT and CONFIG queries are validated by Tiers 1 and 2 alone.
The Judge Model
The judge model is Qwen2.5-7B-Instruct Q4_K_M on Ollama Instance 2 — a completely separate model instance from the generation model on Ollama Instance 1. It is given a completely different system prompt that establishes it explicitly as a quality reviewer: "You are a quality reviewer for an enterprise AI system. You are not the assistant. You are an independent reviewer. Your task is to evaluate the provided answer based solely on the provided documentation. Be rigorous, skeptical, and identify problems."
The judge model does not have access to the generation model's internal reasoning or any information beyond what is explicitly provided in its evaluation prompt. The separation of generation and judgment is essential: if the generation model judged its own output, the judgment would be subject to the same systematic biases that produced the output.
Three Evaluation Dimensions
The judge evaluates three dimensions of answer quality, each scored 0-1.
Faithfulness: Does every specific claim in the answer trace to the retrieved documentation? The judge applies reasoning judgment to assess whether the answer as a whole stays within the bounds of what the documentation says, without extrapolating, generalising, or adding information not present in the retrieved context. This is the model's independent cross-check against the NLI score — if both NLI and the judge find faithfulness issues, the concern is stronger.
Step Completeness: For ERROR_RESOLUTION answers, are all the resolution steps that the documentation says are necessary present in the answer? The judge reads the cause-resolution chunk from the retrieved context and compares its resolution steps against the steps present in the answer. If the documentation shows 5 steps for cause 1 and the answer only mentions 3, this is a completeness failure. The judge identifies specifically which steps are missing.
Relevance: Does the answer address what the employee specifically asked, not what a similar but different question might have asked? An employee asking "why does VL150 appear when MMBE shows stock" should receive an explanation of the safety stock mechanism — not a generic list of all VL150 causes. The judge assesses whether the answer's scope is appropriately matched to the question's scope.
Tier 3 Outcome
The judge returns three dimension scores and an optional list of specific issues found. These scores contribute to the final ValidationScore ensemble. If the judge's faithfulness score is below 0.70, a correction message is automatically appended to the employee's conversation turn: "One correction to the above: {specific_correction_text}." The original streamed answer remains visible — the correction supplements it rather than replacing it.
Source Freshness Coefficient
After all three tiers have produced their scores, a source freshness coefficient is applied to the ensemble score. The coefficient reflects the reality that documentation verified recently is more reliable than documentation that may reflect an outdated SAP configuration.
The freshness reference date is the oldest last_verified_date among the top 5 retrieved chunks. Using the oldest date is conservative — if any chunk's documentation is potentially stale, the answer should reflect that uncertainty.
The coefficient mapping: documentation verified within the last 90 days earns a coefficient of 1.0 (no penalty). 90-180 days old: 0.95. 180-365 days old: 0.85. Over 365 days old: 0.75. These coefficients reflect the observation that SAP configurations change slowly (annual coefficient 0.75 means 25% penalty — significant but not disqualifying) while recent documentation is trusted fully.
Ensemble Scoring and Confidence Routing
The raw ValidationScore is calculated as:
RawScore = (NLI Support Score × 0.45) + (LLM Judge Faithfulness × 0.30) + (LLM Judge Step Completeness × 0.25)
For SIMPLE_FACT and CONFIG queries where Tier 3 does not run, the Tier 3 components are replaced with 1.0 (full trust, not evaluated) — this effectively makes the NLI score the primary signal.
ValidationScore = RawScore × FreshnessCoefficient
Confidence Badge Routing:
ValidationScore ≥ 0.85: Green confidence badge. Answer delivered cleanly with full attribution panel showing Document ID, Configuration/Error Name, Verified By, Verified Date.
ValidationScore 0.70-0.85: Amber confidence badge. Answer delivered with the note: "This answer is based on available documentation. Some steps may require verification with your SAP team." Attribution panel included.
ValidationScore < 0.70 on first attempt: Targeted regeneration triggered. The regeneration prompt includes the specific unsupported claims from Tier 2 (ValidationResult.unsupported_claims) as explicit correction instructions. If the regenerated response achieves 0.70 or above, it is delivered with the amber badge. If it still scores below 0.70, the escalation response is returned and a mock ticket ARQ task is queued.
Answer Attribution Panel
Delivered with every answer regardless of confidence level. The attribution panel is a structured UI element appearing below the answer text showing: the primary source document ID, the document's full name (configuration name or error description), the verification date, and the person who verified it. For multi-document answers (Mode C retrievals spanning multiple documents), up to three source documents are listed. The confidence badge (green or amber indicator) appears alongside the attribution panel.

PART 4 — Data Architecture and Knowledge Management

Section 8 — Zone D: Data Architecture
Overview
Zone D is the persistent memory of AEGIS. Six distinct data stores serve different access patterns, durability requirements, and query types. Each store is configured precisely for its role — using the right data store for each responsibility is what makes AEGIS performant and reliable rather than using a single general-purpose database for everything.
The six stores are: four Qdrant collections (vector search), OpenSearch (keyword search), PostgreSQL with PgBouncer and read replica (structured relational data), Redis Instance 1 (ephemeral session and cache data), and Redis Instance 2 (persistent task queue data).
Redis Instance 1 — Session Store, Cache, and Rate Limiting
Redis Instance 1 is the fast-access ephemeral store. It holds all data that must be accessed with millisecond latency on every request and that is acceptable to lose on restart.
Configuration: Maximum memory is set to 6GB. The 6GB ceiling is appropriate for a 200-employee system — each session hash is under 100KB, the semantic cache stores embeddings (3KB each for 768-dim vectors) and compressed answer text, and rate limiting counters are tiny integers. 6GB provides headroom for thousands of concurrent sessions. The eviction policy is allkeys-lru — when memory reaches 6GB, the least recently accessed keys are evicted. This is correct: recently active sessions are more valuable than dormant ones. Persistence is disabled — session data and cache entries are ephemeral by design. If Redis Instance 1 restarts, sessions are lost and employees get a fresh conversation, which is acceptable.
Data stored:
Session state hashes at session:{session_id} with 2-hour TTL (all fields described in Section 3).
Semantic cache entries at cache:{embedding_hash} where the hash is a base64-encoded SHA-256 of the query embedding bytes. Each entry stores: query_text (the enriched query), answer_text (the delivered answer), validation_score (float), document_ids (JSON array), created_at (ISO timestamp), embedding_model_version (string: "bge-base-en-v1.5"). TTL is 24 hours.
Rate limiting counters at ratelimit:{user_id}:{minute_epoch} — integer counters with 2-minute TTL.
JWT revocation set at revoked_tokens — Redis Set data structure containing jti values of revoked tokens, each with individual TTL equal to the token's remaining lifetime.
DiagnosticObjects at diagnostic:{session_id} — JSON-serialised DiagnosticObject, 10-minute TTL.
Redis Instance 2 — ARQ Task Queue
Redis Instance 2 holds all ARQ task data. Tasks must survive server restarts, requiring persistence.
Configuration: Maximum memory set to 1GB (task queue data is small — task parameters are lightweight JSON except vision tasks which now use file paths not image bytes). Eviction policy is noeviction — when memory fills, Redis returns an error rather than silently dropping tasks. This ensures task queue overflow is visible and actionable rather than causing silent task loss. Persistence uses AOF (Append-Only File) with everysec fsync — every write is committed to disk within 1 second. AOF rewrite runs hourly to compact the log and prevent unbounded disk growth.
Data stored:
ARQ task queues at arq:queue:{task_type} — Redis List data structures. Each list stores serialised task JSON objects. Task types: vision, audit, feedback_diagnosis, cache_write, knowledge_gap, mock_ticket, nightly_cleanup. Tasks are pushed to the list tail and the ARQ worker pops from the head (FIFO).
ARQ task state hashes at arq:task:{task_id} — each active task's current state: queued/running/completed/failed, retry count, last attempt timestamp, error message if failed.
Dead letter lists at arq:dead_letter:{task_type} — tasks that exhausted their retry budget. Monitored by Prometheus and surfaced in Grafana.
Qdrant — Four Vector Collections
Qdrant is the vector database storing all document chunk embeddings and the semantic cache. It runs as a single instance in the demo (no replica) — if Qdrant becomes unavailable, the circuit breaker activates and AEGIS falls back to OpenSearch-only retrieval.
Collection 1: meridian_errors
Stores all error_guide document chunks. Each point contains: a content named vector (768-dim BGE-base embedding of chunk text), an identity named vector (768-dim BGE-base embedding of the document ID + error code + module string), and a payload with fields: chunk_id (string: "{document_id}:chunk:{index}"), document_id (string), content_type (always "error_guide"), module (string: FI/MM/SD/HR/PP/CO/BASIS), error_code (string, nullable), chunk_type (enum: header/cause_resolution/outcome), chunk_index (integer), total_chunks (integer), cause_number (integer, nullable — only for cause_resolution chunks), transactions (string array), last_verified_date (ISO date string), verified_by (string), chunk_text (the full text of this chunk), embedding_model_version (string: "bge-base-en-v1.5").
HNSW parameters: m=32, ef_construction=200, ef=128. Scalar quantization applied to both named vectors.
Collection 2: meridian_procedures
Stores all procedure document chunks with the same point structure. Payload differences: chunk_type values are procedure_header/procedure_steps/procedure_outcome, error_code is absent, procedure_name (string) is present, phase_name (string, for step chunks) is present, step_range (string: "1-3", for step chunks) is present.
Collection 3: meridian_configs
Stores all config document chunks. Payload differences: chunk_type values are config_overview/config_values/config_navigation, error_code is absent, configuration_name (string) is present.
Collection 4: cache_queries
Stores semantic cache entries for fast cached answer lookup. This collection uses lower-quality HNSW parameters since cache lookup is a threshold check rather than a precision retrieval: m=16, ef_construction=100, ef=64. No scalar quantization (cache is small and query performance matters more than memory). Each point: one vector (768-dim BGE-base embedding of the enriched query text), payload: query_text, answer_text, validation_score, document_ids, created_at, embedding_model_version. The nightly ARQ cleanup task deletes points where created_at is older than 24 hours.
OpenSearch — Keyword Search
OpenSearch provides BM25 keyword search complementing Qdrant's semantic search. It runs as a single-node cluster. The single node uses one shard and one replica shard (data redundancy within the single node). JVM heap is explicitly configured to 2GB in the Docker Compose environment variable OPENSEARCH_JAVA_OPTS: "-Xms2g -Xmx2g" — both min and max set to the same value to prevent heap resizing overhead. Container memory limit is set to 4GB, leaving 2GB for OS overhead, file system cache, and OpenSearch off-heap memory.
The SAP documents index stores all document chunks from all content types (unlike Qdrant which splits by content type). The OpenSearch index mapping includes: chunk_text (analyzed text field using the SAP custom analyzer), document_id (keyword field), content_type (keyword field), module (keyword field), error_code (keyword field with a 2x field boost), chunk_type (keyword field), transactions (keyword array field), last_verified_date (date field), chunk_id (keyword field). The error_code field boost means matches on the error code field score twice as high as matches on the general chunk text.
The custom analyzer pipeline details are specified in Section 5 (Stage 3). The analyzer applies on both indexing and querying, ensuring symmetric tokenization.
PostgreSQL — Structured Relational Data
PostgreSQL is the relational data store for all structured data in AEGIS. PgBouncer sits in front of PostgreSQL for connection pooling. A read replica receives all writes via streaming replication and serves analytical queries.
PgBouncer Configuration: Transaction pooling mode. Pool size: 20 PostgreSQL connections. Maximum client connections: 100 application connections. In transaction pooling mode, a PostgreSQL connection is checked out only for the duration of a database transaction (typically 1-5 milliseconds) and returned to the pool when the transaction commits. This allows 100 concurrent application connections to share only 20 actual PostgreSQL connections with minimal wait time.
PostgreSQL Primary — Operational Schema

documents_registry: document_id (text PK), content_type (enum: error_guide/procedure/config), module (text), transactions (text array), last_verified_date (date), verified_by (text), ingested_at (timestamp), chunk_count (integer), status (enum: active/processing/failed/deprecated), parent_content (bytea — Vault Transit-encrypted JSON containing all template field values for parent context hydration).

PostgreSQL Primary — Config Snapshot Schema
config_snapshot: config_category (text), config_key (text), config_value (text — Vault Transit-encrypted for sensitive values), last_updated_at (timestamp), updated_by (text), notes (text nullable). Composite PK on (config_category, config_key).
The Config Snapshot stores the current state of Sona Comstar's SAP configuration. For example: config_category="posting_periods", config_key="1000:current_period", config_value="05/2026" stores that company code 1000 is currently in posting period 5 of fiscal year 2026.
All Config Snapshot reads from the application use a PostgreSQL REPEATABLE READ transaction isolation level. This prevents the scenario where an IT admin updates multiple snapshot rows simultaneously and an employee query reads some updated rows and some not-yet-updated rows from the same logical update. With REPEATABLE READ, the read transaction sees a consistent snapshot of the database as of the moment it starts — it either sees all of the admin's updates (if they committed before the read started) or none of them (if they started after the read started). The partial-update race condition is eliminated.

Keycloak Schema
Keycloak uses a dedicated keycloak schema within the same PostgreSQL instance. Keycloak creates and manages its own tables within this schema. The application database role has no access to the keycloak schema — Keycloak uses its own dedicated database user. This separation ensures application code cannot accidentally read or write Keycloak's identity tables.
PostgreSQL Read Replica
The read replica receives all write operations from the primary through streaming replication with synchronous_commit = on — writes are acknowledged to the application only after they are confirmed on the replica. This means no data is lost if the primary fails after the replica has acknowledged the write.
Analytical queries (knowledge gap analysis, confidence history aggregation, session quality metrics) all route to the read replica via a separate PgBouncer pool. This prevents complex reporting queries from competing with operational read/write traffic on the primary. The FastAPI application uses two connection strings: one pointing to PgBouncer's operational pool (which routes to the primary) and one pointing to PgBouncer's analytical pool (which routes to the replica).

Backup Strategy (continued)
PostgreSQL backup uses WAL (Write-Ahead Log) archiving. WAL segments are archived to a dedicated backup volume on the server filesystem every time a segment completes (typically every 5-10 minutes of write activity). A full base backup runs hourly using pg_basebackup, storing a complete snapshot of all database files to the backup volume. The combination of hourly base backups and continuous WAL archiving means the system can recover to any point in time within the last hour by replaying WAL segments forward from the nearest base backup. The recovery point objective is under 10 minutes of data loss in the worst case.
Qdrant snapshots are taken every 6 hours using Qdrant's built-in snapshot API. Each snapshot captures the full state of all four collections and their indexes. Snapshots are stored to the backup volume alongside PostgreSQL backups. The recovery point objective for Qdrant is up to 6 hours of ingested documents — in practice, re-ingesting documents is straightforward since the source files remain in the admin portal's upload staging area.
Redis Instance 2 backup uses AOF rewrite, which compacts the append-only log hourly. Since Redis Instance 2 stores only ARQ task queues and task state, the maximum data loss on failure is at most the last hour of task state updates — in practice, active tasks are short-lived and the AOF rewrite cycle rarely coincides with active tasks.
Monthly, a restore drill is performed: a backup is restored to a separate Docker container and the restored data is validated by querying known records. This confirms the backup integrity before it is needed in a real failure scenario.

Section 9 — Zone E: Knowledge Management and Administration
Overview
Zone E is the operational surface of AEGIS — the layer through which the IT admin shapes and improves the system's knowledge base and the quality of its answers. It has two major components: the eleven-stage Document Ingestion Pipeline that transforms uploaded documents into searchable vector data across all four search systems, and the Admin Portal that provides the operational interface for all knowledge management tasks.
Understanding Zone E is critical because the quality of AEGIS's answers is ultimately determined by the quality of the knowledge base. The ingestion pipeline's design choices — how it chunks documents, how it handles updates, how it validates templates — directly determine whether retrieval finds the right information. The admin portal's design choices determine whether the IT team can actually maintain and improve the knowledge base effectively.
Document Ingestion Pipeline — Eleven Stages
The Document Ingestion Pipeline is triggered when the IT admin uploads a document through the Admin Portal. It runs as a synchronous process within FastAPI for the first four stages (fast validation stages that provide immediate feedback to the admin if the document is malformed), then continues asynchronously through the remaining stages (the time-intensive embedding and indexing stages). The admin portal shows a progress indicator with the current stage name while ingestion proceeds.
Stage 1 — File Receipt and Magic Bytes Validation
FastAPI receives the uploaded file as a multipart form submission containing the file bytes and the claimed filename. Before any content is read or parsed, the first 12 bytes of the file — the magic bytes — are extracted and compared against the known signatures for the two supported file types.
PDF files begin with the byte sequence 25 50 44 46 — the ASCII characters %PDF. DOCX files begin with 50 4B 03 04 — the ZIP local file header signature used by all modern Office Open XML formats (DOCX, XLSX, PPTX are all ZIP archives containing XML files). If the magic bytes match neither signature, ingestion halts immediately with a specific error: "Unsupported file format. AEGIS accepts .docx (Word documents) and text-based .pdf files only. The uploaded file has an unrecognised format. Please check the file and try again."
If the magic bytes match PDF, an additional quick check confirms the PDF is not password-protected: the first 1024 bytes of the file are scanned for the /Encrypt PDF dictionary entry. A password-protected PDF returns: "This PDF is password-protected and cannot be processed. Please remove the password and re-upload."
Stage 1 completes in under 50 milliseconds.
Stage 2 — Text Extraction
Text extraction uses different libraries for each file type.
For DOCX files, the python-docx library reads the document's internal XML structure (word/document.xml within the ZIP archive). It extracts all paragraph text in document order. The extraction preserves bold, underline, and heading level formatting as Python-readable markers — **bold_text** for bold, _underlined_text_ for underlined, and heading level prefixes H1:, H2:. These formatting markers are significant because AEGIS template field labels are consistently formatted as bold or underlined text, making them identifiable to the field detector in Stage 3 even before specific field name matching.
For PDF files, pdfplumber reads each page of the PDF and extracts the text layer. pdfplumber extracts text from PDFs that have an embedded text layer — the text that was present in the original digital document when the PDF was created. After extraction, two validation checks run: if the total extracted text length is under 100 characters across the entire document, the document is likely a scanned image PDF with no text layer, and ingestion halts with: "This PDF appears to be a scanned image without extractable text. AEGIS cannot process scanned PDFs. Please convert the content to a Word document using the AEGIS template or use a PDF with selectable text." If the extracted text contains more than 30% garbled characters (detected by counting non-printable ASCII characters), this suggests a corrupted PDF text layer and halts with a specific error.
Stage 2 output: the complete document text as a Python string, with formatting markers preserved.
Stage 3 — Template Field Detection
The field detector reads the extracted text line by line, building a field map: an ordered list of (field_name, start_character_position, end_character_position) tuples identifying each template field and its content boundaries.
Field labels are detected using a multi-strategy approach. Primary detection looks for lines matching the pattern FIELD_NAME: — a string of uppercase letters, underscores, and numbers followed by a colon, appearing at the start of a line or after a recognised separator (a line of equals signs, dashes, or asterisks). Secondary detection looks for bold-formatted text (identified by the **...** markers from Stage 2) that matches the pattern of a field label — often template fields are bold in the source document but may not have the colon format.
The field map is constructed by reading the field labels in order. Each field's content spans from the end of its label line to the start of the next field's label line. The last field's content spans to the end of the document. This gives the pipeline a complete structured view of the document as a set of named fields with their content.
If fewer than 5 field labels are detected across the entire document, the template detection confidence is too low and ingestion halts with: "The uploaded document does not appear to follow the AEGIS template format. Only {N} template fields were detected, which is insufficient. Please use the AEGIS document templates available in the admin portal and ensure all required fields are present."
Stage 4 — Content Type Identification and Validation
The content type is read from the CONTENT_TYPE: field in the field map. If this field is absent, ingestion halts. If the value is not one of the three accepted types (error_guide, procedure, config), ingestion halts with a message naming the found value and listing the three valid options.
The document ID is read from the DOCUMENT_ID: field. Its format is validated against the required pattern: {MODULE}-{TYPE}-{NUMBER} where MODULE is one of {FI, MM, SD, HR, PP, CO, BASIS}, TYPE is one of {ERR, PROC, CFG}, and NUMBER is a three-digit zero-padded integer. If the format is invalid, ingestion halts with a specific message showing the found format and the required format.
All required fields for the detected content type are verified. For error_guide, required fields are: DOCUMENT_ID, CONTENT_TYPE, MODULE, ERROR_CODE, TRANSACTIONS, WHEN_THIS_OCCURS, CAUSE_1 (at minimum one cause block), CAUSE_1_HOW_TO_IDENTIFY, CAUSE_1_RESOLUTION_STEPS, SUCCESS_INDICATOR, ESCALATION_CRITERIA, LAST_VERIFIED_DATE, VERIFIED_BY. For procedure, required fields are: DOCUMENT_ID, CONTENT_TYPE, MODULE, PROCEDURE_NAME, PURPOSE, WHEN_TO_USE, PREREQUISITES, TRANSACTIONS, STEP_1 (at minimum three steps), VERIFICATION_STEPS, LAST_VERIFIED_DATE, VERIFIED_BY. For config, required fields are: DOCUMENT_ID, CONTENT_TYPE, MODULE, CONFIGURATION_NAME, WHAT_THIS_CONTROLS, CHANGE_FREQUENCY, NAVIGATION, CURRENT_VALUES_AT_SONA_COMSTAR, LAST_VERIFIED_DATE, VERIFIED_BY.
Missing required fields produce a detailed error listing exactly which fields are absent: "The following required fields are missing from this document: ESCALATION_CRITERIA, LAST_VERIFIED_DATE. Please add these fields using the template format and re-upload."
The document ID is checked against the documents_registry table. If the document ID does not exist, this is a new document ingestion — a new registry entry will be created at Stage 11. If the document ID exists with status "active", this is an update — Stage 5 (chunk invalidation) will run before proceeding. If the document ID exists with status "deprecated", the admin is warned and asked to confirm reactivation.
Stage 4 output: a validated content type, a validated document ID, an ingestion mode (new or update), and a complete field map.
Stage 5 — Chunk Invalidation (Update Mode Only)
When a document update is detected, all existing data for this document must be removed across all search systems before new data is ingested. Running both old and new chunks simultaneously would produce retrieval results that mix outdated and current information — an employee might receive a resolution from the old version and navigation steps from the new version in the same answer.
The invalidation process runs in four steps. First, the documents_registry table is updated to set status = 'processing' for this document ID, preventing queries from citing the document during the transition. Second, all Qdrant points with document_id == this_document_id are deleted from the three content collections and the sparse collection using Qdrant's filter-based delete API. Third, all OpenSearch documents with document_id == this_document_id are deleted using OpenSearch's delete-by-query API. Fourth, all rows in the document_relationships table with from_document_id == this_document_id are deleted.
The parent_content field in the old documents_registry record is preserved in memory during this stage — it is used as a fallback reference if any subsequent stage fails and the update needs to be rolled back (restoring the old record to active status).
Stage 6 — Template-Aware Field-Level Chunking
Chunking is the most intellectually designed stage of the ingestion pipeline. Instead of splitting documents at arbitrary character or token boundaries (which would split cause descriptions from their resolution steps, procedure phases from their steps, and configuration names from their current values), AEGIS uses template field boundaries as natural chunk boundaries.
Chunking Rules for error_guide Documents:
Chunk 0 — type header: Contains all fields from DOCUMENT_ID through WHEN_THIS_OCCURS. This chunk captures the document's identity and the error's observable symptom. Its identity vector embedding will match queries like "what is VL150" or "tell me about the delivery availability error".
Chunks 1 through N — type cause_resolution: One chunk per cause block. A cause block spans from the CAUSE_{N}: separator through CAUSE_{N}_RESOLUTION_STEPS: content and optionally through CAUSE_{N}_RELATED_CONFIG:. The critical rule: CAUSE_N_HOW_TO_IDENTIFY and CAUSE_N_RESOLUTION_STEPS are never split into separate chunks. They always appear together because the identification description is the semantic context that makes the resolution steps meaningful — a retrieval system that separates them will sometimes return resolution steps without the identification context, confusing the generation model about which scenario the steps apply to.
If a cause-resolution chunk exceeds 500 tokens (common for causes with 6+ resolution steps), it is split at the natural boundary between HOW_TO_IDENTIFY and RESOLUTION_STEPS content with a 75-token overlap: the last 75 tokens of the HOW_TO_IDENTIFY section are duplicated at the start of the RESOLUTION_STEPS sub-chunk. This overlap ensures that when the resolution steps sub-chunk is retrieved independently, it has enough identification context to be correctly interpreted.
Chunk N+1 — type outcome: Contains SUCCESS_INDICATOR, ADMIN_STEPS, ESCALATION_CRITERIA, and all document footer fields (RELATED_ERRORS, LAST_VERIFIED_DATE, VERIFIED_BY).
Chunking Rules for procedure Documents:
Chunk 0 — type procedure_header: Contains PROCEDURE_NAME through PREREQUISITES (including WHEN_TO_USE and TRANSACTIONS). This chunk captures the procedure's identity and applicability context.
Chunks 1 through M — type procedure_steps: Steps are grouped into chunks of 3-4 consecutive steps. The critical rule: phase boundaries are never crossed within a single chunk. If a procedure has Phase 1 (Steps 1-5) and Phase 2 (Steps 6-11), no chunk contains steps from both Phase 1 and Phase 2. A 3-step chunk that would require including Step 5 and Step 6 must instead be split as a 2-step Phase 1 chunk (Steps 4-5) and a separate Phase 2 chunk starting at Step 6. Steps within a chunk that have plant-specific exceptions (for example, "For plant 9000, storage location Y013 must be entered at this step") include those exceptions within the same chunk as the step they modify.
A 75-token overlap is applied at chunk boundaries: the last 75 tokens of chunk M are repeated at the start of chunk M+1. This ensures that if an employee asks about a specific step that happens to be near a chunk boundary, both the preceding context and the step itself are available in the retrieved chunk.
Chunk M+1 — type procedure_outcome: Contains VERIFICATION_STEPS, COMMON_ERRORS_IN_THIS_PROCEDURE, POST_COMPLETION_NOTES, LAST_VERIFIED_DATE, VERIFIED_BY.
Chunking Rules for config Documents:
Chunk 0 — type config_overview: Contains CONFIGURATION_NAME through CHANGE_FREQUENCY. This chunk captures the configuration's identity and business purpose.
Chunk 1 — type config_values: Contains the entire CURRENT_VALUES_AT_SONA_COMSTAR section as a single, unbreakable chunk. This section contains the current configuration values specific to Sona Comstar — the most critical and frequently queried content in any config document. This rule has no exception regardless of length. A config_values section of 600 tokens remains as one chunk. The DeBERTa windowing mechanism in the Validation Engine handles long chunks at validation time without requiring the chunk itself to be split.
Chunk 2 — type config_navigation: Contains NAVIGATION, RELATED_ERRORS, CHANGE_PROCESS, LAST_VERIFIED_DATE, VERIFIED_BY.
Stage 7 — Metadata Envelope Construction
Each chunk receives a complete metadata envelope. The envelope is a Python dictionary containing all payload fields that will be stored in Qdrant and mirrored in OpenSearch: chunk_id (formatted as {document_id}:chunk:{index}), document_id, content_type, module, error_code or configuration_name or procedure_name depending on content type, chunk_type, chunk_index, total_chunks, cause_number (nullable), phase_name (nullable), transactions (array), last_verified_date, verified_by, chunk_text (the full text of this chunk), embedding_model_version (set to "bge-base-en-v1.5" at ingestion time for future embedding model tracking).
The embedding_model_version field is the fix for the long-term embedding versioning concern identified in the gap analysis. When the embedding model is eventually upgraded, the pipeline sets the new version string, and the admin can query Qdrant for all points with the old version string and re-embed them. This makes embedding model upgrades a targeted operation rather than a full collection rebuild.
Stage 8 — Embedding Generation
Each chunk is embedded twice by the BGE Embedding Service.
The content vector is generated by embedding the full chunk_text. The BGE-base-en-v1.5 model processes the chunk text and produces a 768-dimensional floating-point vector representing the semantic meaning of the chunk's content. For chunks near the BGE-base model's maximum input length of 512 tokens, the text is truncated at 512 tokens (approximately 380-420 words) before embedding. Chunks designed by Stage 6 to respect the 500-token maximum are safely within this limit.
The identity vector is generated by embedding the identity string: "{document_id} {error_code_or_config_name_or_procedure_name} {module} {transactions_space_separated}". This string captures the document's concept-level identity. The identity vector is used in Qdrant's named vector search to find chunks whose document concept matches the query entity, complementing the content vector's semantic matching.
BGE Embedding Service inference time: approximately 30-60 milliseconds per chunk for content embedding plus 10-20 milliseconds for identity embedding. For a complete error guide with 6 cause-resolution chunks: approximately 8 chunks × 60ms = 480ms total embedding time.
Stage 9 — Qdrant Storage
Each chunk is stored as a point in the appropriate Qdrant collection determined by the document's content_type. The Qdrant upsert call includes the point ID (set to the chunk_id string), both named vectors (content and identity, each 768-dimensional), and the full metadata envelope as the payload.
For update mode: upsert is used (insert if the point ID doesn't exist, replace if it does). This is safe because the chunk invalidation in Stage 5 already deleted all old points for this document — the upsert will always insert rather than replace.
After all chunks are stored in the content-type collection, a second upsert stores each chunk in the meridian_sparse collection, but since BGE-M3 is not used in the demo, this stage is skipped for the demo configuration. The code for this stage exists and will activate automatically in production when BGE-M3 is enabled through configuration.
Stage 10 — OpenSearch Indexing
Each chunk is indexed in OpenSearch's SAP documents index as a separate document. The indexed document contains all fields from the metadata envelope (for keyword search and filtering) plus the chunk_text analyzed through the SAP custom analyzer.
Entity boosting is applied before indexing: the error_code or configuration_name value (whichever is applicable) is appended three times to the chunk text that goes into the analyzed chunk_text field. This triple repetition increases the BM25 term frequency for the entity, causing BM25 queries that include the entity to rank this document higher than documents where the entity appears only once.
Stage 11 — Knowledge Graph Edge Creation and Registry Update
The final stage creates the Knowledge Graph edges and completes the registry record.
The chunker reads two template cross-reference fields. For error_guide and procedure documents, it reads the RELATED_ERRORS and COMMON_ERRORS_IN_THIS_PROCEDURE fields respectively. For config documents, it reads the RELATED_ERRORS field. From these fields, all Document ID references are extracted using the same {MODULE}-{TYPE}-{NUMBER} pattern regex used in Stage 4. For each extracted Document ID reference that exists in the documents_registry table (cross-referenced to prevent orphan edges), a row is inserted into document_relationships.
The documents_registry record is created (for new documents) or updated (for updates) with status "active", the current ingestion timestamp, the chunk count, and the Vault Transit-encrypted parent content JSON (containing all template field values for parent hydration).
The admin portal's document list refreshes automatically. The newly ingested document appears with status "active" and its chunk count. The IT admin can click through to the document detail view to inspect each chunk's text and verify the chunking boundaries are correct.

Feedback Diagnosis Algorithm
The feedback diagnosis ARQ task runs when an employee submits thumbs-down feedback on an answer. Its purpose is to classify the failure: was the answer wrong because the retrieval found the wrong documentation (retrieval failure), or was the answer wrong because the generation model misused correct documentation (generation failure)? This classification determines the appropriate improvement action.
Algorithm Specification:
Step 1: The ARQ task fetches the feedback_events record for this thumbs-down event. It retrieves the original query text, the answer text, and the retrieved document IDs.
Step 2: The retrieval is re-run using the same enriched query text. The same Qdrant, OpenSearch, and RRF fusion steps run as in the original request, producing a fresh set of top-5 chunks. This re-retrieval ensures the diagnosis reflects the current state of the knowledge base, not a cached state from the original query.
Step 3: The original answer text is decomposed into factual claims using the same rule-based claim decomposer used in Tier 2 validation. Each claim is evaluated against the freshly retrieved chunks using the DeBERTa NLI service with the 350-token windowing mechanism.
Step 4: The average maximum entailment score across all claims is computed. This score measures how well the retrieved chunks support the claims in the original answer.
Step 5: Classification decision. If the average entailment score is below 0.65, the retrieved chunks do not adequately support the answer's claims — this is classified as a retrieval failure. The documentation covering this query is either absent from the knowledge base or not being retrieved effectively. A knowledge gap event is written to the knowledge_gap_events table. The feedback_events record is updated with diagnosis_result: {"type": "retrieval_failure", "avg_entailment": 0.58, "action": "knowledge_gap_created"}.
If the average entailment score is above 0.65 (the chunks did support the answer) but the employee still gave negative feedback, the classification is generation failure — the model had correct documentation but generated incorrect information. The feedback_events record is updated with diagnosis_result: {"type": "generation_failure", "avg_entailment": 0.78, "action": "human_review_queued"}. Section 10 — Zone F: Observability and LLMOps
Overview
Zone F provides the operational intelligence layer for AEGIS. It answers the operational questions that the engineering team needs to answer in production: Is the system healthy? Is the AI quality good? Are there knowledge gaps affecting employees? Are there performance regressions? Zone F is intentionally scoped to what a small intern team can realistically operate — two services (Prometheus and Grafana) plus a quality dashboard embedded in the Admin Portal, rather than a six-service ELK + Jaeger stack.
Prometheus — Metric Collection
Prometheus scrapes a /metrics endpoint from every AEGIS service on a 15-second interval. The metrics endpoint on each service is exposed only on the nexus-obs Docker network — no service exposes metrics to the public network or to each other outside observability.
Infrastructure Metrics — Collected From All Services
aegis_http_requests_total (counter, labels: service, endpoint, status_code) — tracks request rates and error rates per endpoint.
aegis_http_request_duration_seconds (histogram, labels: service, endpoint) — tracks latency distribution with p50, p95, p99 buckets.
aegis_process_memory_bytes (gauge, labels: service) — tracks RAM usage per service for the memory allocation monitoring.
aegis_cpu_usage_seconds_total (counter, labels: service) — tracks CPU consumption.
LLM-Specific Quality Metrics
aegis_token_throughput_per_second (gauge, labels: model_instance) — measures generation speed for each Ollama instance. A drop below 2 tokens/second indicates performance degradation.
aegis_context_window_utilisation_pct (histogram) — measures what fraction of the model's context window each prompt uses. Consistently low values (under 20%) may indicate over-aggressive conversation history truncation. High values (above 80%) may indicate context overflow risk.
aegis_validation_score (histogram, labels: query_type) — tracks the distribution of ValidationScores per query type. The distribution shape shows whether answer quality is improving or degrading over time.
aegis_confidence_badge_total (counter, labels: badge_color) — counts green, amber, and no-badge outcomes. A rising no-badge proportion signals increasing answer quality failures.
aegis_crag_state_total (counter, labels: state) — counts SUFFICIENT, INSUFFICIENT, and SKIPPED CRAG outcomes. Rising INSUFFICIENT rates signal knowledge base coverage gaps.
aegis_retrieval_mode_total (counter, labels: mode) — counts Mode A, B, C activations. Rising Mode C proportion suggests the knowledge base has coverage areas that require deep multi-query retrieval to partially address.
aegis_cache_hit_total (counter, labels: result) — counts hits and misses on the semantic cache. Rising hit rate over time indicates the cache is maturing and reducing model load.
aegis_nli_entailment_score (histogram, labels: claim_type) — tracks DeBERTa entailment score distribution. A shift toward lower scores signals that generated answers are becoming less grounded.
aegis_tier_1_failure_total (counter, labels: check_type) — counts Tier 1 validation failures by type (output_leak/scope_boundary/tcode_policy).
aegis_tier_3_judge_score (histogram, labels: dimension) — tracks judge scores for faithfulness, step_completeness, and relevance separately.
aegis_arq_queue_depth (gauge, labels: task_type) — measures the ARQ task backlog per queue. A growing backlog indicates the worker is falling behind and may need investigation.
aegis_arq_task_failure_total (counter, labels: task_type) — counts ARQ task failures per type. A rising failure rate for audit tasks would indicate a PostgreSQL connectivity issue. A rising failure rate for vision tasks would indicate Ollama Instance 3 problems.
aegis_dead_letter_queue_size (gauge, labels: task_type) — measures the dead letter queue size per task type. Any non-zero value requires investigation.
Grafana — Eight Monitoring Panels
Grafana reads from Prometheus and presents monitoring intelligence through eight dashboard panels. The dashboard is designed for a team that checks it once or twice per day rather than for a dedicated operations team monitoring continuously.
Panel 1 — System Health Overview
A summary row showing four key metrics at a glance: current request rate (requests per minute, 5-minute average), current error rate (percentage of requests returning 4xx or 5xx), p95 request latency (the 95th percentile of total request duration including generation time), and system uptime. Alert thresholds: error rate above 5% triggers amber, above 15% triggers red. This is the first panel checked when assessing overall system health.
Panel 2 — Answer Quality Trend
A time-series chart showing the 24-hour rolling average ValidationScore alongside the 7-day average, plotted over the last 30 days. A declining trend is the primary signal that knowledge base quality or model behaviour has changed negatively. Overlaid with the proportion of green vs amber vs no-badge outcomes as a stacked area chart. Alert: if the 24-hour average ValidationScore falls below 0.75, this is a quality regression requiring investigation.
Panel 3 — CRAG State Distribution
A stacked bar chart showing SUFFICIENT, INSUFFICIENT, and SKIPPED proportions per query type over the last 24 hours. Each query type (ERROR_RESOLUTION, PROCESS, CONFIG, SIMPLE_FACT) appears as a separate bar cluster. High INSUFFICIENT rates for a specific query type directly indicate a knowledge base gap for that category. This panel drives documentation priority discussions.
Panel 4 — Retrieval Performance
Side-by-side latency histograms (p50, p95, p99) for each stage of the retrieval pipeline: Qdrant search time, OpenSearch search time, cross-encoder reranking time, and total retrieval pipeline time (from enriched query to final 5 chunks). Spikes in any component indicate that component's performance has degraded. An unexpected spike in cross-encoder reranking time, for example, would suggest that DeBERTa service load has increased.
Panel 5 — Model Generation Performance
A time series of token throughput (tokens/second) for each Ollama instance over the last 24 hours, with a reference line at the baseline throughput measured during initial deployment. Throughput drops more than 25% below baseline indicate that RAM pressure (from other services consuming more memory than allocated) is causing the OS to swap model weights to disk. Cache hit rate is plotted on a secondary axis — a rising cache hit rate should correlate with reduced Ollama load.
Panel 6 — Validation Pipeline Health
Three sub-panels: Tier 1 failure rate by check type (how often output leaks, scope violations, and T-code policy violations are detected), DeBERTa NLI entailment score distribution (histogram showing how well claims are grounded), and Tier 3 judge score distribution per dimension (faithfulness, step_completeness, relevance). Sustained low faithfulness scores from the judge indicate a generation quality problem even if NLI scores are acceptable.
Panel 7 — Infrastructure Resources
A heatmap of CPU utilisation per service and a heatmap of memory utilisation per service, updated every 15 seconds. Colour scale: green (under 70%), amber (70-85%), red (above 85%). The memory heatmap is the most watched panel — any service approaching its allocated limit is a warning that needs investigation. ARQ queue depth for all task types is shown as a time-series subplot.
Panel 8 — ARQ Background Task Health
Queue depth per task type over time (should stay near zero for all types). Task completion rate (tasks completed per minute). Task failure rate per type. Dead letter queue size. A growing dead letter queue for any task type is an automatic alert condition requiring immediate investigation.
Structured Logging and Trace ID Propagation
Every AEGIS service emits structured JSON logs to stdout. Docker captures these logs and makes them queryable via docker logs {container_name}. For the demo, this is the primary debugging tool. For production, these logs can be forwarded to a log aggregation service without any code changes.
Every log entry includes a trace_id field — the UUID4 generated at the Nginx boundary for each request and propagated through every subsequent operation. When debugging a specific employee complaint, the IT admin can search the logs for the session ID or trace ID to find all log entries from all components related to that specific request, in chronological order. The trace_id allows following a single employee query from Nginx validation through Keycloak JWT verification, through the QIL entity extraction, through Qdrant retrieval, through Ollama generation, through DeBERTa validation, to the final WebSocket delivery — as a coherent sequence rather than scattered entries across multiple service logs.

Section 12 — Deployment Architecture
Docker Compose Service Definitions
The AEGIS demo deployment uses Docker Compose with a single docker-compose.yml file defining all 22 services. Services declare their dependencies using the depends_on directive with condition: service_healthy, ensuring each service starts only after its dependencies are confirmed healthy via their health check commands.
Startup Sequence
The startup sequence is determined by the dependency graph. Following the dependency chain, services start in this order:
Layer 1 (no dependencies): Vault (dev mode — starts and unseals immediately), Redis Instance 1, Redis Instance 2. These three start simultaneously.
Layer 2 (depend on Layer 1): PostgreSQL primary (depends on no other AEGIS service), Qdrant (depends on Vault for certificate), OpenSearch (depends on Vault for certificate). These start in parallel once Layer 1 is healthy.
Layer 3 (depend on Layer 2): PgBouncer (depends on PostgreSQL primary health check passing), Keycloak (depends on PostgreSQL primary — uses it as backend database), PostgreSQL replica (depends on primary — begins streaming replication). Start in parallel once Layer 2 is healthy. Keycloak startup takes approximately 60-90 seconds as it initialises the keycloak schema in PostgreSQL and starts the Java application server.
Layer 4 (depend on Layer 3): BGE Embedding Service, DeBERTa NLI Service. These are pure compute services that load model weights and expose inference APIs. BGE-base loads in approximately 10 seconds. DeBERTa loads in approximately 15 seconds. They start once PgBouncer is healthy (confirming the full database path is functional).
Layer 5 (depend on Layer 4): Ollama Instance 1 (Qwen2.5-32B), Ollama Instance 2 (Qwen2.5-7B), Ollama Instance 3 (Qwen2.5-VL-7B). These start simultaneously and each pulls (or loads from local storage) its model. Loading 32B at Q4_K_M takes approximately 2-4 minutes from disk into RAM. The 7B models load in approximately 30-60 seconds. All Ollama instances are configured with OLLAMA_KEEP_ALIVE=-1. Layer 5 is the longest step in the startup sequence.
Layer 6 (depend on Layer 5): FastAPI (2 uvicorn workers) and ARQ worker. Both start once all three Ollama instances are healthy. FastAPI performs startup checks: fetches Keycloak JWKS public keys, authenticates to Vault and retrieves PostgreSQL credentials, verifies Qdrant connectivity, verifies OpenSearch connectivity, verifies Redis connectivity. ARQ worker authenticates to Vault and connects to Redis Instance 2.
Layer 7 (final): Nginx. Starts once FastAPI is healthy. Only after Nginx is running can employees access the system. This ensures no requests reach a partially-initialised application.
Prometheus and Grafana start alongside Layer 2 — they have no functional dependencies and can begin scraping metrics from available services immediately.
Total cold-start time: approximately 5-8 minutes, dominated by Qwen2.5-32B model loading time.

Volume Definitions
vault-data: Vault development mode data (in-memory in dev mode, this volume holds the dev mode root token configuration).
postgres-data: PostgreSQL primary data directory. This volume holds all AEGIS operational and analytical data, all Keycloak user accounts, and all Config Snapshot values.
postgres-replica-data: PostgreSQL replica data directory.
qdrant-data: Qdrant vector index files for all four collections.
opensearch-data: OpenSearch index files and WAL.
redis1-data: Redis Instance 1 (no AOF in demo, but the volume holds the configuration file).
redis2-data: Redis Instance 2 AOF persistence file. Critical — this volume must not be accidentally deleted because it holds the ARQ task queue state.
ollama-models: Shared volume mounted by all three Ollama instances. Contains the pulled model GGUF files. If this volume already contains the model files from a previous startup, startup time for Ollama is dramatically reduced (loading from RAM-mapped files rather than re-downloading).
backup-data: PostgreSQL WAL archives and pg_basebackup outputs, Qdrant snapshots.
tmp-uploads: The /tmp/aegis_uploads directory, mounted as a shared volume between FastAPI and the ARQ worker so both can access the same uploaded image files.
CPU Thread Allocation
The Xeon E-2278G has 16 physical threads. Thread allocation:
Ollama Instance 1 (Qwen2.5-32B): 10 threads via OLLAMA_NUM_THREAD=10. This is the most CPU-intensive service and deserves the majority of threads. Setting this explicitly prevents Ollama from detecting and using all 16 threads simultaneously, which would starve other services.
Ollama Instance 2 (Qwen2.5-7B): 3 threads via OLLAMA_NUM_THREAD=3.
Ollama Instance 3 (Qwen2.5-VL-7B): 3 threads via OLLAMA_NUM_THREAD=3.
DeBERTa NLI Service: 2 CPU cores via Docker --cpus=2.
BGE Embedding Service: 2 CPU cores via Docker --cpus=2.
OpenSearch: 2 CPU cores via Docker --cpus=2.
PostgreSQL, PgBouncer, Redis, Vault, Keycloak, Nginx, FastAPI, ARQ, Prometheus, Grafana: 1 CPU core each or less, sharing the remaining capacity through OS scheduling.
Note: CPU allocations are managed by the OS scheduler — the numbers above are targets, not hard isolation. When the main model is generating (occupying 10 threads), other services run on the remaining 6 threads through time-slicing.

Section 13 — Production Evolution: Demo to Private Cloud GPU
Overview of What Changes
The transition from demo to production modifies four areas. Everything else — the entire retrieval architecture, the complete validation framework, the full security model, all PostgreSQL schemas, all Docker networks, the admin portal, the observability stack — is identical between demo and production. The same codebase runs in both configurations, switching behaviour through environment variable configuration.
Change 1 — Inference Layer
All three Ollama instances are replaced by a vLLM server deployed on a private cloud GPU instance. The vLLM server is hosted on Azure NC-series (A100 80GB GPU) or AWS p4d.xlarge equivalent. The instance is deployed inside a private VNet (Azure) or VPC (AWS) with no public IP address and no inbound rules from the public internet. The AEGIS on-premise server connects to the vLLM API through Azure Private Link or AWS PrivateLink — traffic travels over the cloud provider's private backbone network and never crosses the public internet.
This architecture satisfies the security requirement: SAP query content never leaves the private network boundary. The vLLM server's API is accessible only from the AEGIS on-premise server's private IP through the private link connection.
Model Upgrades:
Main generation: Qwen2.5-72B-Instruct at AWQ quantization (fits in 40GB VRAM on an A100 80GB, leaving 40GB for KV cache). Quality improvement over the demo's 32B Q4_K_M is substantial — the 72B model has deeper reasoning capability and better instruction following.
Judge model: Qwen2.5-14B-Instruct at AWQ quantization running as a second vLLM instance on the same GPU server.
Vision model: Qwen2.5-VL-72B-Instruct at AWQ quantization — the full-scale vision model for significantly better OCR and visual understanding of complex SAP screens.
Performance Improvements:
vLLM enables continuous batching — multiple employee requests are batched together and processed in a single GPU forward pass. This dramatically increases throughput under concurrent load.
Speculative decoding activates in production using Qwen2.5-0.5B as the draft model. The draft model proposes 4 tokens at once; the 72B main model verifies them in one forward pass. For SAP procedural content where sequences like numbered steps are highly predictable, speculative decoding achieves 1.5-2.5× throughput improvement with no quality loss.
First-token latency: drops from 5-8 seconds (CPU, 32B) to 0.8-1.5 seconds (GPU, 72B AWQ). Total response generation for a 300-token answer: drops from 60-90 seconds to 5-10 seconds. This is the most visible user experience improvement of the production deployment.
Change 2 — Authentication Flow
ROPC is replaced with Authorization Code Flow with PKCE. The login flow changes: instead of the AEGIS frontend handling the employee's credentials directly, clicking "Login" redirects the employee's browser to Keycloak's own login page. The employee enters their credentials directly into Keycloak's form — the AEGIS frontend never touches the raw credentials. Keycloak authenticates, then redirects back to AEGIS with an authorization code. The frontend exchanges this code for tokens using PKCE (Proof Key for Code Exchange), which prevents authorization code interception attacks.
This is a frontend code change only — the backend token validation, revocation, and session handling are unchanged.
Change 3 — High Availability Configuration
BGE-M3 sparse retrieval service is deployed alongside BGE-base. The meridian_sparse Qdrant collection is populated during a re-ingestion run triggered at production launch. OpenSearch BM25 + BGE-base dense + BGE-M3 sparse gives the full tri-modal retrieval stack.
The Qdrant replica is added. Production Qdrant runs with primary + replica, providing automatic read load distribution and manual failover capability in under 5 minutes.
Vault is configured with cloud KMS auto-unseal. Azure Key Vault or AWS KMS holds the unseal key. Every Vault restart (including planned maintenance restarts) auto-unseals without human intervention.
Change 4 — Real Ticket Integration
The ARQ mock_ticket task type is replaced by a real_ticket task type. The real ticket task calls the Sona Comstar IT ticketing system's API (ServiceNow, JIRA, or email gateway depending on what they use). The ticket payload includes all the same fields as the mock ticket plus the employee's anonymised user ID for IT follow-up routing.
The mock_tickets PostgreSQL table is kept for audit purposes — all tickets are still recorded locally. The real ticket task additionally calls the external API.
Change 5 — Infrastructure Scale
Docker Compose is replaced by Docker Swarm for the on-premise components. Docker Swarm provides service replication (multiple FastAPI worker containers behind Nginx), rolling updates (deploy new code versions without downtime by updating one container at a time), and simple multi-node clustering if additional servers are added later. The Docker service definitions in Swarm are nearly identical to the Compose definitions — the migration requires adding deploy: replicas: N to each service and specifying resource limits as Swarm constraints.
